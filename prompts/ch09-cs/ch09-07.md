> **Prompt**: Please explain the difference between C# value types and reference types using memory diagrams.

To understand how C# manages data under the hood, you have to look at how it splits computer memory into two distinct areas: the **Stack** and the **Heap**.

Whether a variable is a **Value Type** or a **Reference Type** completely changes where its data is stored and how it behaves when you copy it.

---

## The Two Memory Areas

* **The Stack:** A fast, small, and automatically managed memory area. It works like a stack of plates—data is allocated sequentially when a method is called and instantly wiped clean when the method exits.
* **The Heap:** A large, flexible pool of memory. It is used for long-lived data or data that needs to dynamically grow. The **Garbage Collector (GC)** is responsible for cleaning up this area when data is no longer needed.

---

## 1. Value Types (Stored on the Stack)

Value types hold their data **directly** within their own memory slot. In C#, all primitive numbers (`int`, `double`, `bool`, `char`), `enum`s, and `struct`s (like `DateTime`) are value types.

### Code Example:

```csharp
int x = 10;
int y = x; // Copying the value
y = 20;    // Changing y does NOT affect x

```

### Memory Diagram:

As you can see in the diagram, `x` and `y` have their own independent slots on the Stack. When you write `int y = x;`, the value `10` is literally duplicated into `y`'s slot. Because they are completely separate copies, changing `y` to `20` has zero impact on `x`.

---

## 2. Reference Types (Stored on the Heap)

Reference types do not hold their actual data directly. Instead, they store a **memory address (a reference/pointer)** that points to the location on the Heap where the real data lives. Classes, `string`s, arrays, and interfaces are reference types.

### Code Example:

```csharp
public class Player { public int Score; }

Player p1 = new Player() { Score = 100 };
Player p2 = p1; // Copying the reference (address)
p2.Score = 500; // Changing p2 DOES affect p1!

```

### Memory Diagram:

Look at what happens here:

1. When `p1` is created, the `new` keyword allocates space for the actual `Player` object out on the **Heap** (let's say at memory address `0xAB`). The variable `p1` on the **Stack** only stores the address `0xAB`.
2. When you write `Player p2 = p1;`, you aren't copying the player. You are copying the *address*. Now, `p2` on the stack also stores `0xAB`.
3. Because both `p1` and `p2` point to the exact same object on the Heap, changing `p2.Score` to `500` instantly changes the score when you read it from `p1`. They are two remotes controlling the exact same television.

---

## Summary Comparison

| Feature | Value Types | Reference Types |
| --- | --- | --- |
| **Examples** | `int`, `double`, `bool`, `struct` | `class`, `string`, `array`, `interface` |
| **Primary Location** | **Stack** | **Heap** (Reference is on the Stack) |
| **What is copied?** | The actual data value | The memory address pointer |
| **Assignment Effect** | Independent copies | Shared reference to the same object |
| **Memory Cleanup** | Instant (when method ends) | Handled later by the **Garbage Collector** |

Understanding this distinction is the key to avoiding subtle bugs where you accidentally modify shared data, and it helps you write much more memory-efficient C# code.
