> **Prompt**: Compare switch statements and switch expressions in C#.

While they share a name and a similar core purpose—branching code based on a value—**switch statements** and **switch expressions** in C# represent two entirely different programming paradigms.

The **switch statement** (available since C# 1.0) is *procedural* and focuses on **executing actions**.
The **switch expression** (introduced in C# 8) is *functional* and focuses on **computing a value**.

---

## 1. Syntax Comparison

The difference in intent becomes immediately obvious when you look at the boilerplate code required for each.

### The Switch Statement (The Classic Way)

A switch statement is built out of control blocks. It uses `case` labels, requires a `break` (or `return`/`goto`) to exit the block, and relies on a `default:` label.

```csharp
// Focus: Execute code blocks
switch (errorCode)
{
    case 400:
        message = "Bad Request";
        LogWarning(message);
        break; // Mandatory boilerplate
    case 404:
        message = "Not Found";
        break;
    default:
        message = "Unknown Error";
        break;
}

```

### The Switch Expression (The Modern Way)

A switch expression treats the operation as a single math-like equation. It strips away `case` and `break`, uses the **`=>` (lambda/fat arrow) operator**, and uses the **`_` (discard)** symbol instead of `default`.

```csharp
// Focus: Evaluate and return a value directly
string message = errorCode switch
{
    400 => "Bad Request",
    404 => "Not Found",
    _   => "Unknown Error" // The discard replaces 'default'
};

```

---

## 2. Key Structural Differences

| Feature | Switch Statement | Switch Expression |
| --- | --- | --- |
| **Primary Goal** | **Control Flow:** Run a block of statements. | **Value Assignment:** Compute and return a result. |
| **Boilerplate** | High (`case`, `:`, `break;`). | Minimal (`=>`, `,`). |
| **Can execute multiple lines?** | Yes, natively inside the case block. | No, limited to single expressions (or a block that calls a method). |
| **Exhaustiveness** | Optional. If no case matches, it just skips the block. | **Mandatory.** If no pattern matches, the program throws a runtime exception. |
| **Fall-through (`case 1: case 2:`)** | Supported natively by stacking cases. | Handled via pattern matching logic (`1 or 2 => ...`). |

---

## 3. Deep Dive: Exhaustiveness and Safety

One of the biggest advantages of switch expressions is **compiler safety**.

If you use a switch statement on an enum and forget a value, the compiler doesn't care. It will silently skip past the switch block, which often leads to subtle bugs down the line.

```csharp
enum Role { Admin, Editor, Guest }

// If a new role "Manager" is added later, this statement silently does nothing
switch (role)
{
    case Role.Admin:   GrantFullAccess(); break;
    case Role.Guest:   GrantReadOnly();   break;
}

```

With a switch expression, **the compiler demands completeness**. If you don't map out every single possibility (or provide a default `_` discard), the Roslyn compiler will flag it immediately with a compilation warning (`CS8509: The switch expression does not handle all possible values`).

If you ignore the warning and a missing value hits the expression at runtime, .NET will instantly throw an `InvalidActivePatternException` rather than letting your application enter a corrupt or undefined state.

## 4. Multiple Labels vs. Pattern Matching

When you need to map multiple inputs to the same outcome, the code architecture diverges completely.

### Stacking in Statements

Switch statements achieve this by letting execution "fall through" empty case labels:

```csharp
switch (day)
{
    case DayOfWeek.Saturday:
    case DayOfWeek.Sunday:
        isWeekend = true;
        break;
    default:
        isWeekend = false;
        break;
}

```

### Pattern Combining in Expressions

Switch expressions leverage C#'s powerful logical pattern matching (`or`, `and`) right on the line:

```csharp
bool isWeekend = day switch
{
    DayOfWeek.Saturday or DayOfWeek.Sunday => true,
    _ => false
};

```

---

## Summary Recommendation

> **Use a Switch Expression** as your default choice whenever you are translating an input into an output (e.g., converting enums to strings, calculating pricing tiers, state transitions). It is cleaner, less error-prone, and safer.
> **Use a Switch Statement** only when you aren't trying to return a value, but instead need to execute complex multi-line procedural code blocks, open file streams, or use loop controls (`break`, `continue`) inside the individual case branches.
