> **Prompt**: Please explain short-circuit evaluation in C# with practical examples.

**Short-circuit evaluation** is a clever optimization rule used by compilers and runtimes when evaluating boolean expressions (logical `AND` and `OR`).

Instead of evaluating every single condition in a statement from left to right, the runtime **stops evaluating as soon as the final outcome of the expression is mathematically guaranteed.**

Think of it as an "early exit" for logic checks. It saves CPU cycles and, more importantly, protects your code from crashing.

---

## 1. How It Works Under the Hood

The mechanics are based entirely on the truth tables of `AND` (`&&`) and `OR` (`||`).

### The Logical AND (`&&`) Short-Circuit

For an `AND` expression to be true, **all** parts must be true ($A \text{ and } B = \text{True}$).

* If the compiler evaluates the first condition and finds it is **False**, the entire expression is guaranteed to be false.
* There is no point in looking at the rest of the conditions, so the runtime completely skips them.

### The Logical OR (`||`) Short-Circuit

For an `OR` expression to be true, only **one** part needs to be true ($A \text{ or } B = \text{True}$).

* If the compiler evaluates the first condition and finds it is **True**, the entire expression is already guaranteed to be true.
* The runtime immediately short-circuits and skips evaluating the rest.

---

## 2. Practical Examples

Short-circuiting is incredibly useful in real-world programming. Here are two classic scenarios where it shines.

### Example 1: The "Crash Shield" (Preventing Null Exceptions)

The most common use of short-circuit evaluation is checking if an object exists before trying to read its properties.

Imagine you want to check if a user's name starts with "A", but the `user` object itself might be `null`.

```csharp
// Without short-circuiting (if using a single '&'), this crashes if user is null!
if (user != null && user.Name.StartsWith("A"))
{
    Console.WriteLine("User authorized.");
}

```

* **If `user` is null:** The first expression (`user != null`) evaluates to **False**. Because we are using `&&`, the runtime stops right there. It never reads `user.Name.StartsWith("A")`.
* **Why this matters:** If it *did* try to read the second part, the application would instantly throw a `NullReferenceException` and crash. Short-circuiting acts as a defensive shield.

### Example 2: The "Performance Saver" (Skipping Heavy Operations)

You can use short-circuiting to avoid running slow, resource-heavy code (like database queries or API calls) if a simpler check fails first.

```csharp
bool IsAdmin(User user)
{
    // 1. Fast memory check vs 2. Slow database check
    return user.HasAdminFlag || CheckDatabaseForAdminPermissions(user.Id);
}

```

* **If the user has the flag:** `user.HasAdminFlag` is **True**. Because we are using `||`, the runtime realizes the condition is satisfied. It short-circuits and **never runs** the expensive `CheckDatabaseForAdminPermissions` function.
* **Why this matters:** Your application avoids a wasted network trip to the database, making your code run much faster for administrators.

## 3. The Dangerous Side Effect: Missed Code

Because short-circuiting skips code, you must be careful if the right side of your expression contains a function that *changes state* (has a side effect).

Look at this tricky scenario:

```csharp
bool isExpired = true;
int goldCoins = 10;

// The function increments goldCoins by 5 and returns true
bool AwardBonus() {
    goldCoins += 5;
    return true;
}

// CRITICAL MISTAKE: Because isExpired is true, the OR short-circuits!
if (isExpired || AwardBonus())
{
    Console.WriteLine("Checking account status...");
}

Console.WriteLine(goldCoins); // Prints 10, NOT 15!

```

Because `isExpired` was already `true`, the `||` operator skipped `AwardBonus()` entirely. The user never gets their coins, creating a subtle logic bug.

> **Rule of Thumb:** Keep the left side of your logical operators reserved for quick, safe checks (like null handling or boolean flags), and avoid putting crucial state-changing logic on the right side of a short-circuit operator.
