> **Prompt**: Please explain reflection with simple practical examples and show how dependency injection frameworks use reflection.

In traditional C#, your code is static and predictable. You write a class, you instantiate it using the `new` keyword, and you call its methods. The compiler knows exactly what types exist before your application even runs.

**Reflection** is a powerful feature in .NET that flips this model upside down. It allows an application to inspect its own metadata, explore available types, and dynamically instantiate objects or invoke methods **at runtime**, even if those types didn't exist when the program was compiled.

Think of reflection as giving your program a magnifying glass to look at its own compiled blueprint (the assembly manifest).

---

## 1. A Simple Practical Example of Reflection

To use reflection, you start with the `System.Type` class, which acts as the gateway to an object's metadata.

Imagine you have a simple class, and you want to inspect it dynamically:

```csharp
public class Laptop
{
    public string Model { get; set; }
    public void Boot() => Console.WriteLine("Laptop is booting up...");
}

```

Using reflection, we can inspect and interact with this class completely on the fly:

```csharp
using System;
using System.Reflection;

// 1. Get the "Type" metadata object
Type laptopType = typeof(Laptop);

Console.WriteLine($"Class Name: {laptopType.Name}");

// 2. Discover fields/properties dynamically
PropertyInfo[] properties = laptopType.GetProperties();
foreach (var prop in properties)
{
    Console.WriteLine($"Found Property: {prop.Name} ({prop.PropertyType.Name})");
}

// 3. Instantiate the object dynamically (Equivalent to: new Laptop())
object laptopInstance = Activator.CreateInstance(laptopType);

// 4. Find and invoke a method dynamically
MethodInfo bootMethod = laptopType.GetMethod("Boot");
bootMethod.Invoke(laptopInstance, null); // Prints: "Laptop is booting up..."

```

### Why would you do this?

In a normal app, you wouldn't. Writing `new Laptop()` is faster and type-safe. However, reflection is invaluable when building **generic tools**. For example, a JSON serializer uses reflection to loop through an unknown object's properties at runtime and convert them into text.

## 2. How Dependency Injection (DI) Frameworks Use Reflection

Modern .NET applications rely heavily on **Dependency Injection (DI)** containers (like the built-in `Microsoft.Extensions.DependencyInjection` or Autofac).

When you request a service via a constructor, you never write the `new` keyword yourself. The DI container miraculously hands you a fully constructed object with all of its dependencies satisfied. **Reflection is the engine that makes this possible.**

### The Scenario

Imagine you have a controller that requires an order service, which in turn requires a database connection:

```csharp
public interface IDatabase { }
public class SqlDatabase : IDatabase { }

public class OrderService
{
    // OrderService DEPENDS on IDatabase
    public OrderService(IDatabase database) { } 
}

```

You register these types with the DI container at startup:

```csharp
var services = new ServiceCollection();
services.AddTransient<IDatabase, SqlDatabase>();
services.AddTransient<OrderService>();

var provider = services.BuildServiceProvider();

```

### Behind the Scenes: The DI Reflection Loop

When you eventually ask the provider for the service (`provider.GetService<OrderService>();`), the DI framework doesn't have a hardcoded `new OrderService(new SqlDatabase())` statement. Instead, it executes a reflection-based algorithm:

```csharp
// Conceptually, this is what the DI Container does behind the scenes:

Type targetType = typeof(OrderService);

// 1. Use reflection to inspect the object's constructors
ConstructorInfo[] constructors = targetType.GetConstructors();
ConstructorInfo primaryConstructor = constructors[0];

// 2. Use reflection to look at the required parameters
ParameterInfo[] parameters = primaryConstructor.GetParameters();

List<object> resolvedDependencies = new List<object>();

foreach (ParameterInfo param in parameters)
{
    Type dependencyType = param.ParameterType; // e.g., IDatabase
    
    // 3. Look up what class maps to this interface in the container registry
    Type implementationType = LookUpRegisteredType(dependencyType); // Returns SqlDatabase
    
    // 4. Recursively instantiate the dependency using reflection
    object dependencyInstance = Activator.CreateInstance(implementationType);
    resolvedDependencies.Add(dependencyInstance);
}

// 5. Construct the final object by passing the resolved dependencies array
object finalService = primaryConstructor.Invoke(resolvedDependencies.ToArray());

```

Without reflection, a DI container would be blind. It wouldn't be able to read constructor parameters or match them against registered interfaces dynamically.

## 3. The Double-Edged Sword: The Cost of Reflection

While reflection enables incredible frameworks like DI, object mappers (AutoMapper), and testing suites (xUnit), it comes with two severe trade-offs:

1. **Performance Overhead:** When you use reflection, the CLR must search string-based metadata tables, verify security access checks, and unbox types. This is significantly slower than executing compiled native machine code instructions.
> *Note: Modern DI frameworks solve this by using reflection **only once** at startup to build a dependency graph, caching the creation plan in memory to avoid repeating the performance hit.*


2. **Loss of Compile-Time Safety:** If you pass a typo string to reflection (`laptopType.GetMethod("B00t")`), the compiler won't catch it. Your code will compile perfectly fine but instantly throw a `NullReferenceException` at runtime when it tries to run the missing method.

### Modern Alternative: Source Generators

Because of the performance and Ahead-Of-Time (AOT) compilation limitations of reflection, modern .NET (introduced heavily in recent versions) is moving toward **Source Generators**. Instead of inspecting code at runtime using reflection, Source Generators inspect code *during compilation* and automatically generate the necessary boilerplate C# files before the app even runs, giving you the best of both worlds.
