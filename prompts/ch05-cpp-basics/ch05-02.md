> **Prompt**: Please show me three real-world bugs caused by C-style strings and how `std::string` prevents them.

Transitioning from C-style strings (`char*`) to `std::string` is one of the most effective ways to harden your code. C-style strings are essentially raw pointers to memory with a "promise" that a null terminator (`\0`) exists somewhere at the end. When that promise is broken, systems fail.

Here are three common real-world bugs caused by C-style strings and how C++ handles them.

---

## 1. The "Off-by-One" Buffer Overflow
This is the classic security vulnerability. In C, you must manually ensure that the destination buffer is large enough to hold the source string **plus** the null terminator.

**The C Bug:**
```c
char buffer[10];
// If "username" is 10 characters, strcpy will write 11 bytes (including \0).
// This overflows the buffer and can overwrite the return address on the stack.
strcpy(buffer, "UserTwelve12"); 
```

**How `std::string` prevents it:**
`std::string` manages its own memory dynamically. When you assign a value, it automatically reallocates enough space to fit the new data. You don't have to calculate sizes or worry about the null terminator.
```cpp
std::string buffer;
buffer = "UserTwelve12"; // Automatically resizes; no overflow possible.
```

---

## 2. The Missing Null Terminator (Heartbleed Style)
If a string is created without a `\0` at the end, functions like `printf` or `strlen` will keep reading memory until they happen to find a zero byte. This can leak sensitive data stored in adjacent memory.



**The C Bug:**
```c
char secret[5] = {'P', 'a', 's', 's', 'w'}; // No null terminator!
printf("%s", secret); // Will print "Passw" followed by random memory "garbage".
```

**How `std::string` prevents it:**
The `std::string` class stores the length of the string as a member variable. It doesn't rely solely on finding a `\0` to know when to stop. Furthermore, it handles the null termination internally and automatically, ensuring that whenever you access the underlying data, it is valid.

---

## 3. Dangling Pointers and Use-After-Free
In C, strings are often passed around as pointers. If the function that "owns" the string frees that memory, but another part of the program still holds the pointer, you get a crash or a security hole.

**The C Bug:**
```c
char* getName() {
    char name[] = "Gemini";
    return name; // BUG: returning address of a local stack variable!
}

char* ptr = getName();
printf("%s", ptr); // CRASH: ptr points to memory that has been reclaimed.
```

**How `std::string` prevents it:**
`std::string` follows **Value Semantics**. When you return a `std::string`, the data is moved or copied to the caller. The caller then owns its own copy of the data, so there is no risk of the memory being pulled out from under it.
```cpp
std::string getName() {
    return "Gemini"; // Returns a full object, not a pointer to local memory.
}

std::string name = getName(); // Perfectly safe.
```

---

### Comparison of Safety Features

| Feature | C-style (`char*`) | C++ `std::string` |
| :--- | :--- | :--- |
| **Sizing** | Manual; prone to error | Automatic |
| **Termination** | Manual `\0` required | Handled internally |
| **Memory Ownership** | Ambiguous (who calls `free`?) | Clear (RAII managed) |
| **Copying** | `strcpy` (unsafe) | Simple `=` operator |
| **Length Lookup** | $O(n)$ (must count bytes) | $O(1)$ (stored value) |
