> **Prompt**: Give me 10 common mistakes beginners make when switching from C to C++.

Switching from C to C++ often feels like moving from a manual transmission to a spacecraft. While the syntax looks identical, the underlying philosophy is worlds apart. Many beginners fall into the "C with Classes" trap, treating C++ as just a slightly more organized version of C.

Here are 10 common mistakes to watch out for:

### 1. Using `malloc` and `free` instead of `new` and `delete`
In C, `malloc` only allocates raw memory. In C++, `new` does two things: it allocates memory **and** calls the object's constructor. Using `malloc` on a C++ class means your object is never properly initialized, leading to undefined behavior.
* **The Fix:** Use `new` and `delete`, or better yet, smart pointers.

### 2. Using C-Style Arrays instead of `std::vector` or `std::array`
Fixed-size arrays like `int arr[10]` are prone to buffer overflows and don't know their own size. C++ containers provide safety, automatic memory management, and iterator support.
* **The Fix:** Default to `std::vector` for dynamic sizes and `std::array` for fixed sizes.

### 3. Neglecting the Standard Template Library (STL)
Beginners often waste time writing custom linked lists or sorting algorithms. The STL is highly optimized and battle-tested.
* **The Fix:** Lean on `<algorithm>`, `<vector>`, and `<map>`. Don't reinvent the wheel unless you're doing it for a specific learning exercise.

### 4. Overusing `void*` for "Generic" Programming
In C, `void*` is the go-to for polymorphism. In C++, this bypasses type safety and is generally discouraged. 
* **The Fix:** Use **Templates**. They allow for generic code that remains type-safe and often performs better due to compile-time optimizations.

### 5. C-Style Casting
C-style casts `(int)myFloat` are "sledgehammer" casts—they tell the compiler to "just make it work," which can hide dangerous conversions.
* **The Fix:** Use C++ named casts: `static_cast`, `const_cast`, `reinterpret_cast`, or `dynamic_cast`. They are more specific and easier to find during a code search.

### 6. Ignoring References (`&`)
C programmers tend to pass everything by pointer. While this works, C++ references are often cleaner and safer because they cannot be null (under normal circumstances) and don't require dereferencing syntax (`*` or `->`).
* **The Fix:** Use **const references** (`const T&`) to pass large objects into functions without copying them.

### 7. Manual Resource Management
Manually calling `delete` is a recipe for memory leaks, especially when exceptions are thrown.
* **The Fix:** Embrace **RAII** (Resource Acquisition Is Initialization). Use smart pointers like `std::unique_ptr` and `std::shared_ptr` so that memory is automatically freed when it goes out of scope.

### 8. Using `#define` for Constants
Macro constants like `#define PI 3.14` have no type and no scope, which can lead to frustrating naming collisions.
* **The Fix:** Use `constexpr` or `const` variables.
    ```cpp
    constexpr double pi = 3.14159;
    ```

### 9. Not Understanding Destructors
In C, you manually clean up before a function ends. In C++, objects have lifetimes. If you allocate memory in a constructor but forget to write a destructor to free it, you'll have a leak every time an object is destroyed.
* **The Fix:** Always define a destructor if your class manages a resource, or follow the **Rule of Zero** by using smart pointers.

### 10. Mixing `stdio.h` and `iostream`
While `printf` and `scanf` still work, mixing them with `std::cout` and `std::cin` can lead to synchronization issues and confusing code.
* **The Fix:** Stick to `iostream` for standard C++. It is type-safe and extensible (you can teach `cout` how to print your custom classes).

### Summary Table

| Feature | C Way (Avoid) | C++ Way (Preferred) |
| :--- | :--- | :--- |
| **Allocation** | `malloc` / `free` | `new` / `delete` or Smart Pointers |
| **Strings** | `char*` or `char[]` | `std::string` |
| **Constants** | `#define` | `constexpr` or `const` |
| **Casting** | `(int)x` | `static_cast<int>(x)` |
| **Arrays** | `int arr[5]` | `std::array<int, 5>` or `std::vector` |
