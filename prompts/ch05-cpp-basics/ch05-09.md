> **Prompt**: Please explain how overload resolution works in C++ with tricky edge cases.

Overload resolution is the process the C++ compiler uses to decide which function to call when multiple functions have the same name. Think of it as a **multi-stage elimination tournament** where the compiler tries to find the "Best Viable Function."

If the compiler finds one clear winner, the code compiles. If it finds two or more equally good candidates, you get an **Ambiguity Error**. If it finds none, you get a **No Match Error**.

---

## 1. The Three Stages of Selection

### Stage 1: Name Lookup
The compiler finds all functions with the matching name in the current scope. This includes functions found via **Argument-Dependent Lookup (ADL)**.

### Stage 2: Identifying Viable Functions
A function is "viable" if:
* It has the correct number of parameters (or has defaults/variadics).
* There is a way to convert the arguments provided to the types of the parameters.

### Stage 3: Finding the Best Match (Ranking)
The compiler ranks the conversions required for every argument. The function that requires the "cheapest" conversions for all arguments—and is strictly better in at least one—wins.

---

## 2. The Ranking Hierarchy (The "Cheapest" Wins)

The compiler uses a strict hierarchy to rank how it converts your data:

1.  **Exact Match:** No conversion, or trivial conversions (e.g., Array-to-Pointer, adding `const`).
2.  **Promotion:** Small types moving to larger types that can't lose data (e.g., `char` to `int`, `float` to `double`).
3.  **Standard Conversion:** Conversions that might lose data (e.g., `int` to `double`, `double` to `int`, or pointer conversions).
4.  **User-Defined Conversion:** Using a constructor or a conversion operator defined in a `class`.
5.  **Ellipsis (`...`):** Variadic functions are the "choice of last resort."

---

## 3. Tricky Edge Cases

### A. The "Best-at-Everything" Rule
To win, a function must be **at least as good** as every other candidate for every argument, and **strictly better** for at least one.

```cpp
void func(int a, double b); // (1)
void func(double a, int b); // (2)

func(10, 10); // ❌ Error: Ambiguous
```
* For argument 1, (1) is an Exact Match, but (2) is a Standard Conversion. **(1) wins.**
* For argument 2, (2) is an Exact Match, but (1) is a Standard Conversion. **(2) wins.**
* Since neither is better at *everything*, the compiler gives up.



### B. Pointer vs. `0` vs. `NULL` vs. `nullptr`
In older C++, `0` and `NULL` were often just integers. This caused major headaches.

```cpp
void log(int i);    // (1)
void log(char* c);  // (2)

log(NULL);    // In some compilers, this calls (1) because NULL is 0.
log(nullptr); // ✅ Correct: Calls (2) because nullptr is a pointer type.
```

### C. `const` Overloading
If you have a member function, you can overload it based on the `const`-ness of the object calling it.

```cpp
class Database {
public:
    void query() { std::cout << "Mutable access"; }       // (1)
    void query() const { std::cout << "Read-only access"; } // (2)
};

void run(const Database& db) {
    db.query(); // Calls (2) because the reference is const.
}
```

### D. Templates vs. Non-Templates
If a non-template function and a template function are both "Exact Matches," the **non-template function always wins**. The compiler assumes that if you wrote a specific version, you meant to use it over the generic version.

```cpp
template<typename T> void print(T t); // (1)
void print(int t);                    // (2)

print(10); // Calls (2).
```

---

## 4. The "Pitfall": Integral Promotion
A common trick in C++ exams involves `bool`, `char`, and `short`.

```cpp
void trap(int x);  // (1)
void trap(bool x); // (2)

trap(10); // Calls (1)
trap(1);  // Calls (1) ... 1 is an int literal!
```
Even though `1` could be a `bool`, it is an `int` literal. Since (1) is an Exact Match for an `int`, it is chosen over the conversion to `bool`.
