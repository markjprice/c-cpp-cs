> **Prompt**: Please show me some common pointer bugs in C and how to fix them.

Pointer bugs are the leading cause of crashes ("Segmentation Faults") and security vulnerabilities in C. Because C gives you direct access to memory, it assumes you are a perfect bookkeeper. When that bookkeeping fails, the results are often unpredictable.

Here are the four most common "hall of fame" pointer bugs and the professional ways to solve them.

---

## 1. The Dangling Pointer
A dangling pointer occurs when you have a pointer that still holds the memory address of a block that has already been deleted or freed. 

* **The Bug:** You `free()` a pointer but forget that the pointer variable itself still contains the address. If you try to access it again, you are touching "dead" memory.
* **The Risk:** Use-after-free vulnerabilities, which hackers use to take control of programs.

```c
char *name = malloc(20);
strcpy(name, "Gemini");
free(name); 

// BUG: name still holds the address!
printf("%s", name); // Undefined behavior/Crash
```

### The Fix: Reset to NULL
Always set a pointer to `NULL` immediately after freeing it. In C, dereferencing `NULL` will cause an immediate, predictable crash rather than a silent, exploitable corruption.

```c
free(name);
name = NULL; // Now the pointer is "safe"
```

---

## 2. Memory Leaks
A leak happens when you lose the "key" to a room in the heap. If you overwrite a pointer before freeing the memory it points to, that memory is lost to the system until the program ends.



* **The Bug:** Re-assigning a pointer without freeing the old allocation.

```c
int *data = malloc(100);
data = malloc(200); // BUG: The first 100 bytes are now unreachable!
```

### The Fix: Proper Lifecycle Management
Ensure every `malloc` has exactly one corresponding `free`.

```c
int *data = malloc(100);
// ... use data ...
free(data);        // Clear old space
data = malloc(200); // Now safe to reuse the variable
```

---

## 3. Buffer Overflow
This happens when you write more data into a memory block than it was designed to hold. Since C doesn't check bounds, it will happily overwrite whatever variables happen to be sitting next to your buffer in memory.

* **The Bug:** Using "unsafe" functions like `gets()` or `strcpy()` that don't check lengths.

```c
char buffer[10];
// BUG: Writing 20 chars into a 10-char box
strcpy(buffer, "This string is way too long!"); 
```

### The Fix: Bounds-Checked Functions
Use functions that require a maximum length, like `strncpy()` or `fgets()`.

```c
// Only copies up to 9 chars + null terminator
strncpy(buffer, "This string is long", sizeof(buffer) - 1);
buffer[sizeof(buffer) - 1] = '\0'; 
```

---

## 4. Uninitialized Pointers (Wild Pointers)
When you declare a pointer, it isn't empty; it contains whatever "garbage" values were left in that memory spot. This is a "Wild Pointer."

* **The Bug:** Dereferencing a pointer before assigning it an address.

```c
int *p; // Contains a random address like 0x3A21...
*p = 10; // BUG: Writing 10 to a random spot in the OS!
```

### The Fix: Initialize to NULL or a Value
Never leave a pointer "naked." Always give it a value or set it to `NULL` at the moment of creation.

```c
int *p = NULL; // Safe
// ... later ...
p = &some_var;
```
