> **Prompt**: Please show me how a large C program becomes hard to maintain as it grows beyond five files.

When a C project expands from a handful of files to a massive codebase, it typically hits a "complexity wall." C lacks the built-in modularity features of modern languages (like namespaces or package managers), so maintaining order requires manual discipline that often breaks down under pressure.

Here is how the architecture begins to crumble as the file count grows.

---

## 1. The "Include Hell" (Circular Dependencies)
In a small project, you might have `main.c` and a few helpers. As you hit 10+ files, your headers start including other headers. This leads to a tangled web where changing a single struct in `common.h` forces a recompilation of the entire project.

* **The Problem:** You accidentally create a circular dependency (File A includes B, which includes A).
* **The Result:** Compilation errors that are notoriously difficult to trace, or "header bloat" where every file includes 50 unnecessary definitions.

## 2. Namespace Pollution
C has a flat namespace for global symbols. In a five-file project, it’s easy to remember you used `init()` for the logger. In a fifty-file project:
* The Graphics team writes `void init()`.
* The Networking team writes `void init()`.
* **The Result:** The linker throws a "multiple definition" error. Developers end up using messy prefixes like `NET_SOCKET_V4_init()` just to avoid collisions, making the code verbose and ugly.

## 3. Global State Side Effects
Small C programs often rely on `extern` variables to share data. As the project grows, these globals become "dark matter"—state that can be changed from anywhere at any time.

| Scaling Factor | Impact on Maintenance |
| :--- | :--- |
| **5 Files** | You know exactly where `global_config` is modified. |
| **50 Files** | A bug appears where `global_config` is corrupted. You must now grep dozens of files to find which pointer arithmetic went wrong. |

## 4. Opaque Pointer Breakdown
To keep files decoupled, C developers use **Opaque Pointers** (forward declarations). While this helps "hide" implementation details, as the project grows, it becomes harder to track the **ownership** and **lifecycle** of these objects. 
* Who is responsible for `free()`? 
* If File A passes a pointer to File B, which passes it to File C, the "contract" of who owns that memory often gets lost in the growth, leading to leaks or use-after-free bugs.

## Summary of the "Entropy"
In a small project, you hold the mental map in your head. Beyond five files, you rely on the **linker** and **preprocessor**. Because C gives you zero "safety rails" for how files interact, the code naturally trends toward a "Big Ball of Mud" unless you strictly enforce architectural patterns (like Pimpl or strict layering) from day one.
