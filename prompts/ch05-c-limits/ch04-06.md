> **Prompt**: Please explain why global variables are dangerous in large C programs with a real example.

In a small C program, a global variable is a convenient "shortcut." In a large program, a global variable is a **side-effect landmine**.

The danger lies in the fact that global variables break **temporal coupling**—the assumption that a variable’s state remains consistent between the time you check it and the time you use it. Because any function in any file can modify a global, you lose the ability to reason about your code locally.

---

## The Real-World Example: The "Ghost" Command
Imagine you are building a flight control system for a drone. You use a global variable to track the current "Operating Mode."

### The Setup
In `globals.h`:
```c
// 0 = Manual, 1 = Autonomous, 2 = Emergency Land
int G_SystemMode = 0; 
```

### The Conflict
Two different modules interact with this global variable at different frequencies.

**Module A: Navigation (`nav.c`)**
This module checks the mode to decide if it should execute a high-speed turn.
```c
void execute_navigation() {
    if (G_SystemMode == 1) { // Check: Are we in Autonomous?
        // ... Logic takes 5 milliseconds ...
        perform_high_speed_maneuver(); // Action
    }
}
```

**Module B: Safety Monitor (`safety.c`)**
This module monitors battery levels. It can trigger an interrupt or run on a separate thread.
```c
void monitor_battery() {
    if (battery_critically_low()) {
        G_SystemMode = 2; // Force Emergency Land mode
        stop_all_motors();
    }
}
```

### The "Race Condition" Corruption
The danger occurs when the **Safety Monitor** changes the global variable *after* the **Navigation** module has checked it, but *before* it has finished its action.

1.  **Nav** checks `G_SystemMode`. It is `1` (Autonomous).
2.  **Safety** detects a low battery. It sets `G_SystemMode` to `2` (Emergency).
3.  **Nav** (unaware of the change) proceeds to `perform_high_speed_maneuver()`.
4.  **Result:** The drone attempts a high-stress turn while the battery is dying, causing a mid-air power failure and a crash.

## Why This Is Harder to Maintain as You Grow

### 1. The "Grep" Nightmare
If `G_SystemMode` ends up with an invalid value (like `-1`), and your project has 500 files, you have to search the entire codebase for every instance where that variable is assigned. In an encapsulated system, you would only have to check one "Setter" function.

### 2. Testing Isolation is Impossible
To "Unit Test" the Navigation module, you cannot simply test the function. You must also set up the global state of the entire system. This makes tests "brittle"—a change in the Safety module can suddenly cause the Navigation tests to fail, even though the Navigation code hasn't changed.

### 3. Hidden Dependencies
Global variables create "invisible threads" between files. You might think `nav.c` and `safety.c` are independent, but they are physically linked through the memory address of that global. If you try to reuse `nav.c` in a different drone project, it won't compile because it's "addicted" to a global variable that doesn't exist in the new project.

---

## The Professional Fix: "Getters and Setters"
Instead of a raw global, teams use a **Static File-Scope Variable** with access functions.

```c
// system_state.c
static int system_mode = 0; // Private to this file

void set_system_mode(int mode) {
    // We can add "Guard Logic" here
    if (mode < 0 || mode > 2) return; 
    
    // We could add a Mutex/Lock here for thread safety
    system_mode = mode;
}

int get_system_mode() {
    return system_mode;
}
```

By forcing all modules to go through `set_system_mode()`, you create a "chokepoint" where you can log changes, set breakpoints, and prevent invalid data from entering the system.
