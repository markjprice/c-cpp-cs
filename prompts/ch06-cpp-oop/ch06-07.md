> **Prompt**: How does polymorphism affect performance in C++?

Polymorphism in C++ is a classic example of the "abstraction penalty." While it provides immense flexibility, it comes with specific costs in terms of memory, CPU cycles, and compiler optimization.

In C++, performance is affected by two main types of polymorphism: **Static** (Templates) and **Dynamic** (Virtual Functions).

---

## 1. Dynamic Polymorphism (The Runtime Cost)
Dynamic polymorphism relies on the **vtable** (Virtual Table) system. Every call to a `virtual` function incurs three distinct performance hits.

### A. Memory Overhead (Size)
Every class with virtual functions gets one `vtable` (a static array of function pointers). More importantly, every **instance** of that class grows by the size of a pointer (the `vptr`), typically **8 bytes** on 64-bit systems. 
* **The Impact:** If you have millions of small objects (like `Point` classes), adding `virtual` can double or triple your memory usage, potentially leading to more cache misses.

### B. Indirection (The "Triple Jump")
When you call a virtual function, the CPU cannot jump directly to the code. It must:
1.  Load the object's `vptr`.
2.  Lookup the function address in the `vtable`.
3.  Jump to that address.



### C. The "Optimization Killer" (Inlining)
This is often the most significant cost. Compilers love to **inline** small functions (copying the function code directly into the caller) to save overhead. 
* Because the compiler doesn't know which version of a virtual function will be called until runtime, it **cannot inline the call**. 
* This prevents downstream optimizations like constant folding or instruction scheduling that usually happen after inlining.

---

## 2. Static Polymorphism (The Compile-Time Cost)
Static polymorphism uses **Templates**. This shifts the "cost" from the CPU to the compiler.

* **Runtime Performance:** Usually **zero overhead**. Since the type is known at compile time, the compiler can inline the code and optimize it just like a regular function.
* **Binary Size:** Can lead to **"Code Bloat."** If you use a template with 10 different types, the compiler generates 10 different versions of that function in your executable.
* **Compile Time:** Significantly increases the time it takes to build your project.

---

## 3. CPU Branch Prediction
Modern CPUs try to "guess" which way a branch (like an `if` statement or a function jump) will go. 
* If you call the same virtual function on a list of objects that are all the same type, the CPU's **Branch Target Buffer (BTB)** will predict the jump correctly, and the overhead is negligible.
* If the list contains a random mix of different derived types, the CPU will constantly "mispredict" the jump, forcing it to throw away work and restart the pipeline. This can result in a **10x–20x slowdown** compared to a predicted jump.



---

## Performance Comparison Table

| Feature | Static (Templates) | Dynamic (Virtual) |
| :--- | :--- | :--- |
| **Call Overhead** | None (Inlined) | Pointer indirection |
| **Memory Cost** | $0$ bytes per object | $+8$ bytes per object |
| **Binary Size** | Increases (Code Bloat) | Minimal increase |
| **Optimization** | Full compiler visibility | Limited (No Inlining) |
| **Flexibility** | Fixed at Compile-time | Changeable at Runtime |

---

## When should you care?
For 90% of applications, the cost of a virtual function is so small (a few nanoseconds) that it doesn't matter. However, you should avoid virtual functions in **"Hot Loops"**—sections of code that run millions of times per second (e.g., inside a physics solver, a pixel shader, or a high-frequency trading engine).

In those cases, developers often use **"Data-Oriented Design"** or **CRTP** (Curiously Recurring Template Pattern) to get the benefits of polymorphism without the runtime vtable overhead.
