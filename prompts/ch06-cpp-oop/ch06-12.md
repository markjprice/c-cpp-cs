> **Prompt**: Please explain `const` correctness in C++ with practical examples.

**Const correctness** is a design philosophy in C++ where you use the `const` keyword to explicitly declare your intentions about which data can be modified and which cannot. 

Think of it as a **contract** between you and the compiler: you promise not to change certain values, and in return, the compiler promises to catch you (at compile-time) if you accidentally break that promise.

---

## 1. Const Variables (The Basics)
In C, `const` was often used just for global constants. In C++, it’s used everywhere to prevent "accidental mutation."

```cpp
const int max_speed = 120;
// max_speed = 150; // ❌ Compiler error: cannot assign to a variable that is const
```

---

## 2. Const and Pointers
This is the most confusing part for beginners. The position of `const` relative to the asterisk (`*`) determines what is protected.

* **Pointer to Const:** You can change where the pointer points, but not the value it points to.
    ```cpp
    const int* ptr = &x; 
    // *ptr = 10; // ❌ Error
    ptr = &y;     // ✅ OK
    ```
* **Const Pointer:** You can change the value, but the pointer is stuck pointing at one address.
    ```cpp
    int* const ptr = &x;
    *ptr = 10;    // ✅ OK
    // ptr = &y;  // ❌ Error
    ```
* **Const Pointer to Const:** Everything is locked.
    ```cpp
    const int* const ptr = &x; // 🔒 Complete lockdown
    ```



---

## 3. Const in Function Parameters
This is the "bread and butter" of const correctness. By passing objects by **const reference**, you get the speed of a pointer (no copying) with the safety of a value (no accidental changes).

```cpp
void printVector(const std::vector<int>& v) {
    // v.push_back(10); // ❌ Error: The compiler protects 'v'
    std::cout << v.size(); // ✅ OK: size() is a read-only operation
}
```
**Why it's practical:** If you don't use `const`, other developers (and the compiler) have to assume your function might change the data, which prevents many optimizations.

---

## 4. Const Member Functions
In a class, you must mark functions that don't modify the object's state as `const`. If you don't, you won't be able to call those functions on a `const` instance of your class.

```cpp
class Player {
    int health = 100;
public:
    // This is a "Getter" - it should be const
    int getHealth() const { 
        return health; 
    }

    // This is a "Setter" - it cannot be const
    void takeDamage(int damage) {
        health -= damage;
    }
};

void checkStatus(const Player& p) {
    std::cout << p.getHealth(); // ✅ OK: getHealth is marked const
    // p.takeDamage(10);        // ❌ Error: Cannot call non-const method on const object
}
```



---

## 5. The "Mutable" Escape Hatch
Sometimes an object is *logically* const but needs to change a small piece of internal data for bookkeeping (like a cache or a mutex). You can use the `mutable` keyword for this.

```cpp
class Database {
    mutable int access_count = 0;
public:
    void query() const {
        // Technically modifying the object, but allowed because of 'mutable'
        access_count++; 
        /* perform query */
    }
};
```

---

## Why bother with Const Correctness?

| Benefit | Description |
| :--- | :--- |
| **Documentation** | It tells other programmers "this function will not touch your data." |
| **Security** | It prevents accidental state changes in complex logic. |
| **Optimization** | The compiler can make better assumptions about memory, leading to faster code. |
| **Correctness** | It forces you to think about the "ownership" and "mutability" of your data. |

---

### Pro-Tip: "Const-All-The-Things"
A common practice in modern C++ is to make every variable `const` by default. Only remove the `const` if you find that you actually need to change the value later. This leads to much safer code that is easier to debug.
