> **Prompt**: Please explain how vtables work internally in C++ for virtual functions.

The **vtable** (Virtual Method Table) is the mechanism C++ uses to support **dynamic dispatch**—deciding which function to call at runtime rather than compile time.

While the C++ standard doesn't strictly dictate how polymorphism must be implemented, almost every major compiler (GCC, Clang, MSVC) uses a vtable/vptr system.

---

## 1. The Two Components
To make virtual functions work, the compiler creates two hidden structures:

### A. The vtable (The "Map")
A static array created for **each class** that contains virtual functions. It is essentially a look-up table of function pointers. There is only one vtable per class, regardless of how many objects you create.

### B. The vptr (The "Pointer")
A hidden pointer added to **each object instance**. It points to the vtable of the class that the object actually belongs to.



---

## 2. How it Works Step-by-Step

Imagine we have the following hierarchy:

```cpp
class Base {
public:
    virtual void func1() { std::cout << "Base 1"; }
    virtual void func2() { std::cout << "Base 2"; }
};

class Derived : public Base {
public:
    void func1() override { std::cout << "Derived 1"; }
    // Note: func2 is NOT overridden
};
```

### At Compile Time:
1.  **Base vtable:** The compiler creates a table for `Base` containing pointers to `Base::func1` and `Base::func2`.
2.  **Derived vtable:** The compiler creates a table for `Derived`. Since `func1` is overridden, it points to `Derived::func1`. Since `func2` is NOT overridden, it points back to the original `Base::func2`.

### At Runtime (The Lookup):
When you write `ptr->func1();`, the CPU performs three steps:
1.  Follow the **vptr** inside the object to find the **vtable**.
2.  Look up the correct **index** in the vtable (the compiler knows `func1` is at index 0).
3.  **Jump** to the address found at that index and execute the code.



---

## 3. The Performance Cost
Because of this "triple-jump," virtual functions have a small overhead:
* **Memory:** Each object is slightly larger (usually 8 bytes on a 64-bit system) because it has to carry the `vptr`.
* **Latency:** It takes slightly longer to call a virtual function because the CPU has to fetch the vtable address and then fetch the function address (two memory indirections).
* **Optimization:** Inlining becomes much harder for the compiler because it doesn't know which function will be called until the code is actually running.

---

## 4. Key Behaviors to Remember

### The "Slicing" Trap
If you pass an object by **value** instead of by **pointer/reference**, the `vptr` is "sliced" off. The new object will only have the `Base` vtable, and polymorphism will stop working.

### Why Constructors Can't Be Virtual
When a `Base` constructor is running, the `Derived` part of the object hasn't been built yet. The `vptr` currently points to the `Base` vtable. If you called a virtual function in the constructor, it would call the `Base` version, even if the object is actually a `Derived`.

### Virtual Destructors
If you don't mark a destructor as `virtual`, it isn't added to the vtable. When you `delete` a `Derived` object through a `Base*`, the compiler will just look at the `Base` type, see no virtual destructor entry, and call only the `Base` destructor—leaving the `Derived` memory leaked.
