> **Prompt**: Please give me some examples of bad class design in C++ and how to refactor them.

Bad class design in C++ often stems from "C thinking" (ignoring encapsulation) or over-engineering (excessive inheritance). In C++, bad design doesn't just make code hard to read—it often leads to memory leaks and undefined behavior.

Here are three common examples of bad design and how to refactor them.

---

## 1. The "God Class" (Blob)
A "God Class" is one that tries to do everything. It manages its own data, handles file I/O, performs business logic, and manages UI state. This violates the **Single Responsibility Principle (SRP)**.

### Bad Design:
```cpp
class Player {
public:
    // Mixed responsibilities
    void updatePhysics();
    void drawToScreen();
    void saveToDisk();
    void playSound();
private:
    float x, y;
    std::vector<uint8_t> textureData;
    std::string savePath;
};
```
**The Problem:** If you change how your graphics work, you shouldn't have to touch the `Player` class logic. This design is brittle and impossible to unit test.

### Refactored Design (Composition):
Break the class into smaller, focused components and compose them within the `Player` class.



```cpp
struct Transform { float x, y; };
class Renderer { public: void draw(const Transform& t); };
class SaveSystem { public: void save(const Transform& t); };

class Player {
public:
    void update() { /* logic only */ }
private:
    Transform position;
    Renderer renderer; // Component
    SaveSystem saver;  // Component
};
```

## 2. Public Data Members (The "Struct" Trap)
Beginners often leave data members `public` to avoid writing getters and setters. This is fine for a simple `struct` (Plain Old Data), but dangerous for a `class` that manages state.

### Bad Design:
```cpp
class BankAccount {
public:
    double balance; // Anyone can set balance = -999999.0;
};
```

**The Problem:** You lose **Invariants**. An invariant is a condition that must always be true for an object (e.g., "balance cannot be negative"). If the data is public, the class cannot guarantee its own integrity.

### Refactored Design (Encapsulation):
Make data `private` and provide controlled access.

```cpp
class BankAccount {
private:
    double balance = 0.0;
public:
    void deposit(double amount) {
        if (amount > 0) balance += amount;
    }
    double getBalance() const { return balance; } // Note the 'const'!
};
```

## 3. The "Fragile Base Class" (Deep Inheritance)
Using inheritance to share code rather than to represent an "Is-A" relationship leads to rigid hierarchies.

### Bad Design:
```cpp
class FileHandler {
protected:
    void openFile();
    void closeFile();
};

// LogManager is NOT a FileHandler; it just USES one.
class LogManager : public FileHandler { 
public:
    void log(std::string msg) { 
        openFile(); 
        /* write msg */ 
        closeFile(); 
    }
};
```

**The Problem:** If `FileHandler` changes its constructor requirements, `LogManager` breaks. This is "Implementation Inheritance" and it creates tight coupling.

### Refactored Design (Composition/Dependency Injection):
Instead of inheriting, pass the handler in or keep it as a member.



```cpp
class LogManager {
private:
    FileHandler& fileHandler; // Reference to the tool it needs
public:
    LogManager(FileHandler& fh) : fileHandler(fh) {}
    void log(std::string msg) {
        fileHandler.write(msg);
    }
};
```

## 4. Manual Memory Management (Pre-RAII)
Managing pointers with manual `new` and `delete` inside a class is the most common cause of memory leaks in C++.

### Bad Design:
```cpp
class Buffer {
private:
    int* data;
public:
    Buffer(size_t size) { data = new int[size]; }
    ~Buffer() { delete[] data; }
    // BUG: If you copy this class, both copies point to same memory!
};
```

### Refactored Design (Rule of Zero):
Avoid manual pointers entirely by using standard containers or smart pointers.



```cpp
class Buffer {
private:
    std::vector<int> data; // Handles its own memory perfectly
public:
    Buffer(size_t size) : data(size) {}
    // No destructor, copy constructor, or assignment operator needed!
};
```

---

### Summary Table: Design Anti-Patterns

| Bad Pattern | Why it's bad | Refactoring Goal |
| :--- | :--- | :--- |
| **God Class** | Too many responsibilities | Use **Composition** (SRP) |
| **Public Data** | No control over state | Use **Encapsulation** (Private) |
| **Deep Inheritance** | Tight coupling | **"Favor Composition over Inheritance"** |
| **Manual Pointers** | Memory leaks/Crashes | Use **RAII** (`std::vector`, Smart Pointers) |
