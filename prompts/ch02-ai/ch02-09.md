> **Prompt**: Compare how memory is handled in C, C++, and C# using the same simple example (for example, allocating an array). Explain what actually happens in memory in each case.

To understand the difference between C, C++, and C#, it helps to visualize memory as a giant grid of cubby holes. Each language approaches how it "claims" and "cleans" these holes differently.

We will use the example of **allocating a dynamic array of 10 integers.**

---

## 1. C: The "Manual Mechanic"
In C, memory management is primitive and explicit. You ask the operating system for a raw block of bytes, and you are entirely responsible for it.

### **The Code**
```c
int *arr = (int*)malloc(10 * sizeof(int));
// ... use the array ...
free(arr);
```

### **What happens in memory?**
1.  **Allocation:** `malloc` searches the **Heap** for a contiguous block of 40 bytes (10 integers × 4 bytes each).
2.  **The Pointer:** It returns the address of the very first byte. The variable `arr` is just a number (a memory address) stored on the **Stack**.
3.  **No Initialization:** The cubby holes are **not cleaned**. They contain "garbage data" (whatever was left there by the last program).
4.  **Deallocation:** When you call `free(arr)`, the OS marks those 40 bytes as "available" again. If you forget to call `free`, those bytes stay "locked" until the program closes (a **Memory Leak**).

---

## 2. C++: The "Object-Oriented Architect"
C++ adds a layer of "lifecycle." It doesn't just grab bytes; it "constructs" objects.

### **The Code**
```cpp
int *arr = new int[10];
// ... use the array ...
delete[] arr;
```

### **What happens in memory?**
1.  **The `new` Operator:** Like C, this grabs space on the **Heap**. However, `new` is type-aware. It knows it's making `int`s, not just bytes.
2.  **Construction:** If this were an array of Objects (not just integers), `new` would automatically call the **Constructor** for every single item in the array.
3.  **Type Safety:** You don't have to calculate `sizeof(int)`. The compiler handles the math.
4.  **Strict Deallocation:** You must use `delete[]` (with brackets) for arrays. If you use the standard `delete`, the program might only destroy the first element, leaving the other 9 as a leak.

---

## 3. C#: The "Managed Manager"
C# operates in a "Managed Environment" (the .NET Runtime). You never touch the actual physical address of the memory.

### **The Code**
```csharp
int[] arr = new int[10];
// No "delete" or "free" exists!
```

### **What happens in memory?**
1.  **Managed Heap:** Memory is allocated on the **Managed Heap**.
2.  **Automatic Zeroing:** Unlike C/C++, C# **clears the memory** immediately. Every index in your new array is guaranteed to be `0`.
3.  **The Pointer is a "Handle":** The variable `arr` isn't a fixed memory address. It's a "handle." Why? Because the **Garbage Collector (GC)** might move your array to a different physical location to defragment memory.
4.  **The Garbage Collector:** When your code finishes and `arr` goes out of scope, the memory isn't freed instantly. Instead, the GC periodically scans for memory that no one is using anymore and "recycles" it automatically.

---

## Summary Comparison Table

| Feature | **C (malloc)** | **C++ (new)** | **C# (new)** |
| :--- | :--- | :--- | :--- |
| **Location** | Heap | Heap | Managed Heap |
| **Cleaning** | Garbage data stays. | Garbage data (for primitives). | Auto-zeroed (Clean). |
| **Lifecycle** | Manual (`free`). | Manual (`delete[]`). | Automatic (Garbage Collector). |
| **Safety** | Dangerous (Easy to leak). | Powerful but risky. | High (Safe). |
| **Mental Model** | Managing bytes. | Managing objects. | Managing references. |

### **The "Unreal vs Unity" Context**
* **Unreal (C++):** Even though it's C++, Unreal adds its own **Garbage Collection** on top of it for any class starting with `U` (like `UObject`). If you use standard C++ classes, you're back to manual `new`/`delete`.
* **Unity (C#):** You are almost always in the "Managed" world. However, Unity developers often worry about the GC "hiccup" (a tiny freeze when the GC cleans up memory), which is why we try to avoid creating too many new objects inside an `Update()` loop.
