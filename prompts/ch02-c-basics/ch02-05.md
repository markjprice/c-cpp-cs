> **Prompt**: Please show me how operator precedence works in C with a table and examples that produce surprising results.

In C, **Operator Precedence** determines the "order of operations"—which parts of a mathematical or logical expression are evaluated first. While most people remember **BODMAS/PEMDAS** from school, C has over 15 levels of precedence that can lead to bugs if you aren't careful.

---

## 1. The Precedence Table (High to Low)
This table shows the most common operators. Operators at the top are executed first.

| Precedence | Operator Type | Symbols | Associativity |
| :--- | :--- | :--- | :--- |
| **1 (Highest)** | Postfix | `()`, `[]`, `->`, `++`, `--` | Left-to-Right |
| **2** | Unary | `+`, `-`, `!`, `~`, `(type)`, `*`, `&`, `sizeof` | **Right-to-Left** |
| **3** | Multiplicative | `*`, `/`, `%` | Left-to-Right |
| **4** | Additive | `+`, `-` | Left-to-Right |
| **5** | Relational | `<`, `<=`, `>`, `>=` | Left-to-Right |
| **6** | Equality | `==`, `!=` | Left-to-Right |
| **7** | Logical AND | `&&` | Left-to-Right |
| **8** | Logical OR | `||` | Left-to-Right |
| **9 (Lowest)** | Assignment | `=`, `+=`, `-=`, etc. | **Right-to-Left** |

## 2. Surprising Examples
Here are three classic cases where C's rules might defy your intuition.

### Case A: The Logical Trap
What is the value of `result`?
```c
int a = 5, b = 10, c = 15;
int result = a < b == c > b; 
```
* **Intuition:** It looks like we are checking if `a < b` AND `c > b`.
* **Reality:** Relational operators (`<`, `>`) have higher precedence than Equality (`==`).
    1.  `a < b` is `1` (true).
    2.  `c > b` is `1` (true).
    3.  The expression becomes `1 == 1`.
    4.  **Result:** `1`.

### Case B: Post-Increment vs. Dereference
This is a very common point of confusion when working with pointers.
```c
int arr[] = {10, 20};
int *p = arr;
int val = *p++;
```
* **Intuition:** Does it increment the value at the pointer, or the pointer itself?
* **Reality:** Postfix `++` has higher precedence than prefix `*`.
    1.  `p++` is evaluated first (but because it's *post*-increment, it returns the *old* address to the `*` operator).
    2.  The `*` dereferences the old address.
    3.  **Result:** `val` is `10`, but `p` now points to `20`.



### Case C: The Assignment "Equalizer"
```c
if (x = 0) {
    // This code will never run
}
```
* **Surprise:** This isn't a precedence issue, but a **result** issue. In C, an assignment expression `(x = 0)` returns the value being assigned. Since the value is `0`, the `if` statement sees it as `false`.
