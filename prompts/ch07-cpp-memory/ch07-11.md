> **Prompt**: Why does the Rule of Five exist and when can I ignore it? Please show me the typical performance difference between copy and move operations.

The **Rule of Five** is the modern evolution of the classic C++ "Rule of Three." It exists because C++11 introduced **Move Semantics**, which allows objects to "steal" resources from each other rather than duplicating them.

The rule states that if your class manages a resource (like heap memory, a file handle, or a socket) and you need to define one of the following, you likely need to define all five:

1.  **Destructor** (to release the resource).
2.  **Copy Constructor** (to duplicate the resource).
3.  **Copy Assignment Operator** (to duplicate the resource from an existing object).
4.  **Move Constructor** (to transfer ownership of the resource).
5.  **Move Assignment Operator** (to transfer ownership from an existing object).

---

## 1. Why the Rule of Five Exists
Without the Rule of Five, your objects are "shallow" by default. If your class contains a pointer to the heap:
* **The Copy Problem:** Two objects will point to the same memory. When one dies, it deletes the memory; when the second dies, it tries to delete it again (**Double-Free Crash**).
* **The Performance Problem:** Without move operators, returning a large object from a function forces a full, expensive copy, even if the original is about to be destroyed anyway.



---

## 2. When Can You Ignore It? (The Rule of Zero)
In modern C++, the best practice is actually to **ignore** the Rule of Five by following the **Rule of Zero**.

**The Rule of Zero:** Design your classes so that they don't manage resources directly. Instead of using raw pointers and `new`/`delete`, use standard library containers and smart pointers.

```cpp
// ❌ Rule of Five required: Manages raw pointer
class BadManager {
    int* data;
public:
    BadManager() : data(new int[1000]) {}
    ~BadManager() { delete[] data; }
    // Now you MUST write the other 4...
};

// ✅ Rule of Zero: No special functions needed
class GoodManager {
    std::vector<int> data; // Vector handles its own Rule of Five internally
public:
    GoodManager() : data(1000) {}
    // The compiler-generated destructor/copy/move will work perfectly!
};
```

---

## 3. Performance Difference: Copy vs. Move
The performance gap between a copy and a move depends entirely on the size of the data being managed. 

### The Cost of a Copy ($O(n)$)
When you **copy** a `std::vector<int>` with 1,000,000 elements:
1.  **Allocate** new memory on the heap (time-consuming OS call).
2.  **Iterate** through the original million integers.
3.  **Copy** each integer to the new location.

### The Cost of a Move ($O(1)$)
When you **move** that same vector:
1.  **Copy** the pointer address (8 bytes).
2.  **Copy** the size and capacity integers.
3.  **Set** the original vector's pointer to `nullptr`.



### Typical Benchmarks (Approximate)
| Operation | Size | Time taken |
| :--- | :--- | :--- |
| **Copy** Vector | 1,000,000 ints | **~1.5 - 3.0 milliseconds** |
| **Move** Vector | 1,000,000 ints | **~1 - 5 nanoseconds** |

**The Result:** Moving is roughly **1,000,000 times faster** in this scenario. This is why move semantics transformed C++ performance, especially in high-load systems and game engines.

---

## Summary Checklist
* **Use Rule of Five:** When you are writing a low-level "Resource Wrapper" (like a custom smart pointer or a unique hardware interface).
* **Use Rule of Zero:** For 99% of your application logic. Use `std::string`, `std::vector`, and `std::unique_ptr` as members so you don't have to write a single destructor or copy constructor.
