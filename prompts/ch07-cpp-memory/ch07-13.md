> **Prompt**: Please show me a real example of a memory leak detected with AddressSanitizer.

AddressSanitizer (ASan) is a fast memory error detector for C++. It’s built into most modern compilers (Clang/GCC) and is incredibly effective at catching "invisible" bugs like memory leaks, buffer overflows, and use-after-free errors.

Here is a real-world scenario of a memory leak and exactly what the ASan output looks like when it catches it.

---

## 1. The Buggy Code
Imagine a function that processes a string but fails to free the memory if an early exit condition is met.

```cpp
#include <iostream>
#include <cstring>

void processData(const char* input) {
    // Allocate memory on the heap
    char* buffer = new char[1024]; 
    std::strcpy(buffer, input);

    if (std::strlen(input) > 500) {
        std::cout << "Input too large, aborting.\n";
        return; // 🚩 BUG: Function exits without calling 'delete[] buffer'
    }

    // Normal processing...
    delete[] buffer;
}

int main() {
    // This call will trigger a leak
    processData("A very long string that is actually over 500 characters..."); 
    return 0;
}
```

---

## 2. Compiling with AddressSanitizer
To detect this, you must compile with the `-fsanitize=address` flag (and `-g` for readable line numbers).

**The Command:**
`g++ -fsanitize=address -g leak_example.cpp -o leak_app`

---

## 3. The ASan Output
When you run `./leak_app`, the program will finish, and then ASan will print a "LeakSanitizer" report to your terminal:

```text
=================================================================
==12345==ERROR: LeakSanitizer: detected memory leaks

Direct leak of 1024 byte(s) in 1 object(s) allocated from:
    #0 0x7f8d7a123456 in operator new[](unsigned long) 
    #1 0x55c1d4567890 in processData(char const*) leak_example.cpp:6
    #2 0x55c1d45678da in main leak_example.cpp:18
    #3 0x7f8d79abc123 in __libc_start_main (/lib/x86_64-linux-gnu/libc.so.6+0x21b23)

SUMMARY: AddressSanitizer: 1024 byte(s) leaked in 1 allocation(s).
```



---

## 4. How to Read This Report
ASan gives you a "Backtrace" of the leak:
1.  **Direct leak of 1024 byte(s):** Tells you exactly how much memory was lost.
2.  **#0 operator new[]:** Confirms the memory was allocated using `new[]`.
3.  **#1 processData... leak_example.cpp:6:** This is the most important line. It tells you exactly where the memory was **born**.
4.  **SUMMARY:** Confirms the total damage.

---

## 5. The Refactor (The Fix)
The professional way to fix this isn't just adding a `delete` before the `return`; it's using **RAII** to ensure the memory is freed automatically regardless of how the function exits.

**The Fixed Code:**
```cpp
#include <string>
#include <vector>

void processData(const char* input) {
    // Use std::vector or std::string instead of raw pointers
    std::vector<char> buffer(1024); 
    std::strcpy(buffer.data(), input);

    if (std::strlen(input) > 500) {
        return; // ✅ SAFE: buffer's destructor runs automatically here
    }
} // ✅ SAFE: buffer's destructor runs here too
```



### Why ASan is better than manual debugging
Without ASan, the first code example would run and exit perfectly fine. You wouldn't know there was a leak until your server ran out of RAM three days later. ASan turns a "silent killer" into a loud, obvious error message during development.
