> **Prompt**: Please show me how destructors are called during exception handling in C++.

In C++, the process of cleaning up local objects when an exception is thrown is called **Stack Unwinding**. This is one of the most powerful features of the language because it ensures that resources (like memory, file handles, or mutexes) are not leaked, provided you are using **RAII**.

Here is a step-by-step breakdown of how the C++ runtime handles this.

---

### 1. The Scenario: An Exception is Thrown
Imagine a chain of function calls where each function creates a local object on the stack.

```cpp
#include <iostream>
#include <string>

class Tracker {
    std::string name;
public:
    Tracker(std::string n) : name(n) { std::cout << name << " created\n"; }
    ~Tracker() { std::cout << name << " destroyed\n"; }
};

void funcC() {
    Tracker t3("Object C");
    throw std::runtime_error("Error in C!"); // 🚩 Exception starts here
    std::cout << "This line never runs.\n";
}

void funcB() {
    Tracker t2("Object B");
    funcC();
}

void funcA() {
    Tracker t1("Object A");
    funcB();
}
```

---

### 2. The Process: Stack Unwinding
When `funcC` throws the exception, the C++ runtime begins looking for a `catch` block that can handle it. It searches "up" the call stack (from `C` to `B` to `A`).

As the runtime exits each function to look for a handler, it **destroys all local objects** in that function's scope in the reverse order of their creation.



1.  **In `funcC`:** The runtime sees `t3`. It calls `~Tracker()` for "Object C".
2.  **In `funcB`:** The runtime exits `funcB` and calls `~Tracker()` for "Object B".
3.  **In `funcA`:** The runtime exits `funcA` and calls `~Tracker()` for "Object A".

---

### 3. The Catch Block
Finally, if a `try-catch` block is found in `main`, the exception is handled.

```cpp
int main() {
    try {
        funcA();
    } catch (const std::exception& e) {
        std::cout << "Caught: " << e.what() << "\n";
    }
    return 0;
}
```

**The Console Output would look like this:**
```text
Object A created
Object B created
Object C created
Object C destroyed  <-- Stack Unwinding starts
Object B destroyed
Object A destroyed
Caught: Error in C!
```

---

### 4. Critical Rules for Destructors and Exceptions

#### A. Never Throw from a Destructor
This is the "Golden Rule" of C++ exceptions. If a destructor throws an exception while stack unwinding is *already* in progress (because of a different exception), the C++ runtime will immediately call `std::terminate()`, and your program will crash.
* **Reason:** C++ cannot handle two active exceptions at the same time.



#### B. RAII vs. Raw Pointers
If you use raw pointers and manual `delete`, stack unwinding **will not save you**.

```cpp
void badFunc() {
    int* data = new int[100];
    throw std::runtime_error("Leak!"); 
    delete[] data; // This line is skipped! Memory is leaked.
}
```
If you use `std::vector<int>` or `std::unique_ptr`, their destructors *will* be called during unwinding, and the memory will be freed safely.

---

### Summary Table: Exception Safety

| Event | Result |
| :--- | :--- |
| **Object on Stack** | Destructor is **guaranteed** to run during unwinding. |
| **Object on Heap (Raw)** | Memory is **leaked**; `delete` is skipped. |
| **Object on Heap (Smart Pointer)** | Memory is **freed** via the smart pointer's destructor. |
| **Destructor throws** | **Immediate crash** (`std::terminate`). |

This mechanism is why C++ is often considered more robust for resource-heavy applications than languages that rely on garbage collection, as the cleanup is deterministic and happens the moment it's needed.
