> **Prompt**: Please explain ownership models in C++ with real code examples.

In C++, "Ownership" refers to the responsibility of managing an object's lifetime—specifically, who is responsible for deleting the memory once the object is no longer needed. 

Modern C++ has moved away from manual memory management toward a **Declarative Ownership** model using Smart Pointers. This prevents the "Who calls delete?" ambiguity.

---

## 1. Exclusive Ownership (`std::unique_ptr`)
This is the most common ownership model. It represents a **1-to-1 relationship**: one pointer owns one object. When the `unique_ptr` goes out of scope, the object is automatically destroyed.

* **Constraint:** You cannot copy a `unique_ptr`. You can only **Move** it.



### Code Example:
```cpp
#include <iostream>
#include <memory>

class Engine {
public:
    void start() { std::cout << "Engine roaring!\n"; }
    ~Engine() { std::cout << "Engine destroyed.\n"; }
};

int main() {
    // Allocation: 'p1' owns the Engine
    std::unique_ptr<Engine> p1 = std::make_unique<Engine>();
    
    // std::unique_ptr<Engine> p2 = p1; // ❌ Compile Error: Cannot copy!
    
    // Transferring ownership: 'p2' now owns it, 'p1' is now null
    std::unique_ptr<Engine> p2 = std::move(p1); 
    
    if (!p1) std::cout << "p1 no longer owns the engine.\n";
    p2->start();

} // p2 goes out of scope, Engine is AUTOMATICALLY destroyed here.
```

---

## 2. Shared Ownership (`std::shared_ptr`)
This represents a **Many-to-1 relationship**. Multiple pointers can own the same object. The object is only destroyed when the **last** `shared_ptr` pointing to it is destroyed.

* **Mechanism:** It uses a "Reference Counter" (a small block of memory that tracks how many pointers exist).



### Code Example:
```cpp
#include <iostream>
#include <memory>

void checkTexture(std::shared_ptr<int> ptr) {
    std::cout << "Use count inside function: " << ptr.use_count() << "\n";
}

int main() {
    // Shared ownership of a resource (e.g., a large texture)
    std::shared_ptr<int> tex1 = std::make_shared<int>(500);
    
    {
        std::shared_ptr<int> tex2 = tex1; // Copying is allowed!
        std::cout << "Use count: " << tex1.use_count() << "\n"; // Outputs: 2
        checkTexture(tex1); // Outputs: 3 (temporary copy in function)
    } // tex2 dies, but resource stays alive because tex1 still exists.

    std::cout << "Use count: " << tex1.use_count() << "\n"; // Outputs: 1
} // Last pointer (tex1) dies, resource is destroyed.
```

---

## 3. Non-Owning Observation (`std::weak_ptr` and Raw Pointers)
Sometimes you need to look at an object without owning it.

* **`std::weak_ptr`:** Used specifically with `shared_ptr` to prevent "Circular Dependencies" (where two objects point to each other and never get deleted). It can check if the object still exists before using it.
* **Raw Pointers (`T*`):** In modern C++, raw pointers are used as "observers." They don't call `delete`; they just point to memory owned by someone else.

### Code Example: Circular Dependency Fix
```cpp
struct Node {
    std::shared_ptr<Node> next;
    std::weak_ptr<Node> prev; // If this were shared_ptr, we'd have a leak!
};
```

---

## 4. Value Ownership (The "Default")
Don't forget that the simplest ownership model in C++ is **Value Semantics**. When you declare a variable on the stack, the scope "owns" it.

```cpp
void func() {
    std::string s = "I am owned by this scope";
} // 's' is destroyed immediately here.
```

---

## Summary of Ownership Models

| Model | Tool | Responsibility | Best For... |
| :--- | :--- | :--- | :--- |
| **Exclusive** | `std::unique_ptr` | Single owner | Most local resources, PIMPL pattern. |
| **Shared** | `std::shared_ptr` | Collective ownership | Shared resources (cache, textures). |
| **Observer** | `std::weak_ptr` | No ownership | Breaking cycles, temporary access. |
| **Observer** | `Raw Pointer` | No ownership | Non-owning "views" into data. |



Understanding these models is the key to writing "Modern" C++. By using the correct pointer type, you communicate your architectural intent to both the compiler and other developers.
