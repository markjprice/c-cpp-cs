> **Prompt**: Please show me some real-world examples of ownership transfer using `std::move`.

In C++, `std::move` is the primary way to transfer ownership of a resource from one object to another. It doesn't actually "move" bits; it casts an object to an **rvalue**, telling the compiler: "I don't need this anymore; feel free to steal its internal resources."

Here are three real-world scenarios where this is essential.

---

### 1. Returning a Large Object from a Factory
Imagine you are loading a large 3D model or a high-resolution image. You don't want to copy the massive buffer when returning it from the loader function.

```cpp
#include <vector>
#include <string>

class Mesh {
    std::vector<float> vertices; // Could be millions of floats
public:
    Mesh(std::vector<float> v) : vertices(std::move(v)) {} // Transfer from arg to member
};

Mesh loadModel(std::string path) {
    std::vector<float> rawData = { /* ... load millions of points ... */ };
    
    // Ownership of the heap-allocated vector is moved into the Mesh object.
    // No data is copied; the Mesh just 'takes' the pointer from rawData.
    return Mesh(std::move(rawData)); 
}
```



---

### 2. Passing Ownership to a Thread
When you spawn a background thread to process data, you often want that thread to "own" the data so the original function can exit safely without the data being destroyed.

```cpp
#include <thread>
#include <memory>
#include <iostream>

void processDatabase(std::unique_ptr<std::string> data) {
    std::cout << "Processing: " << *data << std::endl;
} // 'data' is deleted here by the thread

int main() {
    auto myData = std::make_unique<std::string>("Large Sensitive Data");

    // std::unique_ptr CANNOT be copied. We must move it.
    // Ownership is transferred from main() to the new thread.
    std::thread worker(processDatabase, std::move(myData));

    // myData is now nullptr. We cannot use it here.
    worker.join();
}
```



---

### 3. Implementing a "Sink" Function (Unique Resource)
In system programming, you often have a class representing a hardware resource (like a `Socket` or `FileHandle`). You want to ensure only one part of the code "owns" the handle at a time to prevent accidental double-closes.

```cpp
class File {
    int fd; // File descriptor
public:
    File(int f) : fd(f) {}
    ~File() { if (fd != -1) close(fd); }
    
    // Move Constructor: Take the handle from the other File object
    File(File&& other) noexcept : fd(other.fd) {
        other.fd = -1; // Leave the original in a "safe" empty state
    }
};

void uploadToCloud(File f) {
    // This function now exclusively owns the file
} 

int main() {
    File myLog(101);
    
    // Explicit transfer: 'myLog' can no longer be used to write.
    uploadToCloud(std::move(myLog)); 
}
```



---

### Key Takeaway: The "Safe State"
After you call `std::move(obj)`, the object `obj` is left in a **valid but unspecified state**. 

* **For a Vector/String:** It is usually empty (size 0, capacity 0).
* **For a Unique Pointer:** It is always `nullptr`.
* **Rule of Thumb:** Never use an object after it has been moved from, except to assign it a new value or let it go out of scope.

### Comparison Table: Copy vs. Move

| Action | Memory Cost | Logic |
| :--- | :--- | :--- |
| **Copying** | $O(N)$ | Allocates new memory and duplicates every bit. |
| **Moving** | $O(1)$ | Swaps two pointers; no new memory is allocated. |
