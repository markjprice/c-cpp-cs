> **Prompt**: Please show me how two functions could accidentally corrupt shared data using pointers in C.

To understand how C allows accidental corruption, we have to remember that a pointer is just a memory address—a "key" to a house. C doesn't care if you have the right to enter that house, or if two people are trying to renovate the same room at the same time.

Here is a classic scenario involving **Implementation Leakage** and **Shared State**.

---

## The Scenario: The Shared Configuration
Imagine a system where a `GlobalConfig` struct holds the system state. Two different modules, `Networking` and `UI`, both have pointers to this shared data.

### 1. The Vulnerable Structure
In `config.h`, we define our shared data. Because C doesn't enforce encapsulation, any function with the pointer can touch any member.

```c
typedef struct {
    int connection_timeout;
    char *status_message;   // A pointer to a string in memory
} GlobalConfig;
```

### 2. The Conflict
Two functions in different files perform operations that seem safe in isolation but are catastrophic when combined.

**File: networking.c**
The networking module decides the connection is "Offline" and updates the status.
```c
void update_network_status(GlobalConfig *cfg) {
    // This function assumes it owns the string memory
    cfg->status_message = "Offline"; 
}
```

**File: ui.c**
The UI module wants to display a custom greeting and dynamically allocates memory for it.
```c
void refresh_ui_display(GlobalConfig *cfg) {
    // Allocates new memory for a dynamic string
    cfg->status_message = malloc(20);
    strcpy(cfg->status_message, "Welcome User!");
}
```

---

## How the Corruption Happens

The corruption occurs because there is no **ownership contract**.

1.  **UI Module** runs first: It allocates memory on the **Heap** and stores the address in `status_message`.
2.  **Networking Module** runs second: It overwrites that address with a pointer to a **String Literal** (stored in the "Read-Only" section of the binary).
3.  **The Result:** The address of the heap memory is now lost. We have a **Memory Leak**.
4.  **The Crash:** If the UI later tries to `free(cfg->status_message)`, the program will crash because you cannot "free" a string literal.

## Why Encapsulation Would Have Fixed This
If C enforced encapsulation (like a `private` member with a "Setter" function), we could have prevented this:

1.  **Validation:** The setter could check if `status_message` already has memory allocated and `free` it before assigning a new value.
2.  **Read-Only access:** We could provide a `get_status()` function that returns a `const char*`, preventing the UI from accidentally modifying it.
3.  **Thread Safety:** We could hide a **Mutex** inside the setter to ensure two functions don't change the pointer at the exact same microsecond.

Without enforcement, the "correctness" of the code depends entirely on the UI developer and the Networking developer having a meeting to agree on who owns that pointer—a meeting that rarely happens in large, fast-moving teams.
