> **Prompt**: What are common patterns teams use to manage large C codebases?

To prevent a large C codebase from collapsing under its own weight, professional teams move away from "simple C" and toward a set of strict architectural patterns. These patterns emulate the encapsulation and modularity found in higher-level languages.

---

## 1. The Opaque Pointer Pattern (Encapsulation)
This is the closest C gets to a "Class." Instead of exposing the details of a `struct` in a header file, you only provide a forward declaration.

* **The Header (`ship.h`):** Only contains `typedef struct Ship Ship;` and function prototypes.
* **The Source (`ship.c`):** Contains the actual definition of `struct Ship { ... };`.
* **The Benefit:** Other files can hold pointers to a `Ship`, but they cannot access its internal variables directly. This prevents "spaghetti dependencies" where changing one internal variable breaks twenty other files.

## 2. Subsystem Layering (The "Onion" Model)
Teams strictly define which layers of code are allowed to talk to each other. A common hierarchy looks like this:

| Layer | Responsibility | Allowed to Include |
| :--- | :--- | :--- |
| **Application** | High-level logic/UI | Middleware, Drivers |
| **Middleware** | Protocol stacks, File systems | Drivers, HAL |
| **HAL/Drivers** | Direct hardware interaction | Standard Libs |

**The Golden Rule:** A lower layer is *never* allowed to include a header from a higher layer. This prevents circular dependencies.

## 3. The "Unity Build" vs. Incremental Build
As codebases grow, compile times become a major bottleneck. Teams manage this in two ways:
* **Interface-Only Headers:** Keeping headers "clean" (using only forward declarations) so that changing a `.c` file only requires recompiling that specific file.
* **The Unity Build (Alternative):** Some teams actually `#include` multiple `.c` files into one "Master" file. While it sounds messy, it allows the compiler to perform global optimizations and can actually be faster for certain massive projects like *SQLite*.

## 4. Prefix-Based Namespacing
Since C lacks namespaces, teams adopt a rigid naming convention to prevent collisions. Every function and global variable must be prefixed with its module name.
* **Bad:** `init_socket()`, `close_socket()`
* **Good:** `NET_TCP_Init()`, `NET_TCP_Close()`
This makes it immediately obvious where a function originates and prevents the linker from crying "multiple definition."

## 5. X-Macros (Code Generation)
Maintaining large tables of data (like error codes, command lists, or state machines) is error-prone. Teams use "X-Macros" to define a list once and expand it into multiple formats (e.g., an `enum` and a corresponding `string` array).

```c
// List of errors defined once
#define ERROR_LIST \
    X(ERR_NONE, "No error") \
    X(ERR_TIMEOUT, "Request timed out") \
    X(ERR_NOMEM, "Out of memory")

// Expand into an Enum
typedef enum {
#define X(a, b) a,
    ERROR_LIST
#undef X
} ErrorCode;

// Expand into a String Array
const char* ErrorStrings[] = {
#define X(a, b) b,
    ERROR_LIST
#undef X
};
```
