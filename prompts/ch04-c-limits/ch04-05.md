> **Prompt**: Please give me some examples of naming conflicts in C and how developers work around them.

Naming conflicts (or "symbol collisions") occur in C because the linker uses a **flat global namespace**. In a five-file project, you might have one `init()` function; in a project with 500 files and 10 external libraries, you might have twenty.

Here is how these conflicts manifest and the strategies used to survive them.

---

## Common Naming Conflict Scenarios

### 1. The Generic Function Collision
This is the most frequent conflict. Two developers working on different modules choose an obvious name for a common task.

* **Module A (logger.c):** `void log_info(const char* msg);`
* **Module B (math_engine.c):** `void log_info(double value);` (thinking "logarithm info")

**The Consequence:** The code compiles fine individually, but when you link the final executable, the linker stops with a `fatal error: multiple definition of 'log_info'`.

### 2. The Third-Party Library Clash
You include a library for JSON parsing (`json.h`) and another for Networking (`net.h`). Both libraries happen to define a generic `struct Node`.

* **Result:** You cannot include both headers in the same file without a "redefinition" error. Even worse, if they are only defined in the `.c` files, you get a linker error that you cannot easily fix because you don't own the library source code.

### 3. The "Hidden" Global
A developer declares a global variable `int state;` in a file without using the `static` keyword. Because they didn't use `static`, `state` is exported to the global symbol table. If any other file in the entire project also uses a global `state`, the two files will unknowingly **share the same memory location**, leading to bugs that are nearly impossible to track down.

---

## How Developers Work Around Them

Since C doesn't have namespaces (like `Namespace::Function` in C++), teams use "manual namespacing" conventions.

### 1. The Prefix Pattern (The Industry Standard)
Teams agree that every symbol must be prefixed with a short, unique code representing the module.

* **Instead of:** `init()`, `open()`, `close()`
* **Use:** `UI_init()`, `UI_open()`, `UI_close()`



| Module | Prefix | Example Function |
| :--- | :--- | :--- |
| **FileSystem** | `FS_` | `FS_read_block()` |
| **Graphics** | `GFX_` | `GFX_render_frame()` |
| **Network** | `NET_` | `NET_connect_socket()` |

### 2. The `static` Keyword (Internal Linkage)
The `static` keyword is the most powerful tool for "hiding" names. When you declare a function or global variable as `static`, it is only visible within that specific `.c` file.

```c
// In file_a.c
static void internal_setup() { ... } // Safe: Linker won't see this.

// In file_b.c
static void internal_setup() { ... } // Safe: No collision with file_a.
```

Professional C developers make **everything** `static` by default, only "exporting" (removing `static`) the specific functions meant to be part of the public API.

### 3. The "Pseudo-Object" Prefix
To manage structs, developers often combine the prefix with the name of the "object" the function operates on. This mimics Object-Oriented programming.

* **Pattern:** `Prefix_Object_Action`
* **Example:** `AUDIO_Mixer_SetVolume(Mixer *m, int vol);`

### 4. Header Guard Uniqueness
Naming conflicts also happen with macros. To prevent a header from being included twice, developers use "Header Guards." If two headers use the same guard name, the second one will be ignored by the compiler.

* **Bad:** `#ifndef HEADER_H` (Very likely to clash)
* **Good:** `#ifndef PROJECTNAME_MODULE_FILENAME_H`

---

## Summary of Best Practices
| Strategy | Purpose |
| :--- | :--- |
| **Unique Prefixes** | Prevents global collisions across different modules. |
| **Strict `static` Usage** | Keeps the global "symbol footprint" as small as possible. |
| **Opaque Structs** | Prevents name clashes of internal struct members. |
| **Long Macro Names** | Prevents the preprocessor from accidentally replacing common words. |
