> **Prompt**: Please show how namespaces in C++ fix naming conflicts from C.

In C, the only way to avoid naming collisions in large projects is to use long, manual prefixes (e.g., `NET_SOCKET_init`). C++ solves this by introducing **Namespaces**, which allow you to group related code under a logical label. This provides a "walled garden" where the same function names can coexist without interfering with each other.

---

## 1. The Collision: C vs. C++
In C, the linker is "blind" to context—it only sees the function names. In C++, namespaces act as a "first name" for your code.

### The C Way (Collision)
```c
// math_lib.h
void reset() { /* ... */ }

// physics_lib.h
void reset() { /* ... */ } 

// Result: Linker error "multiple definition of 'reset'"
```

### The C++ Way (Namespaces)
```cpp
namespace Math {
    void reset() { /* Resets a matrix */ }
}

namespace Physics {
    void reset() { /* Resets gravity constants */ }
}

// Result: No conflict. You choose which one to call:
Math::reset();
Physics::reset();
```

---

## 2. Hierarchical Organization
Namespaces can be nested, allowing you to create a directory-like structure for your code. This is how large libraries like the C++ Standard Library (`std`) or game engines manage thousands of functions.



```cpp
namespace Engine {
    namespace Graphics {
        class Renderer { /* ... */ };
    }
    namespace Audio {
        class Renderer { /* ... */ }; // No conflict with Graphics::Renderer
    }
}
```

---

## 3. The Scope Resolution Operator (`::`)
The `::` operator tells the compiler exactly where to look for a symbol. This eliminates the "global variable" ambiguity. If you have a local variable and a namespace variable with the same name, you can be explicit:

```cpp
int version = 5; // Global

namespace App {
    int version = 10;
    
    void print() {
        int version = 2;
        std::cout << version;       // Prints 2 (Local)
        std::cout << App::version;  // Prints 10 (Namespace)
        std::cout << ::version;     // Prints 5 (Global)
    }
}
```

---

## 4. Namespace Aliases and "Using"
To avoid typing long paths like `Company::Project::Module::Function()`, C++ provides shortcuts to manage verbosity.

* **Namespace Aliases:**
  `namespace GFX = MyCompany::VeryLongProjectName::Graphics;`
  Now you can just type `GFX::init();`.

* **The `using` Directive:**
  `using namespace std;` 
  This pulls everything from the `std` namespace into the current scope. 
  > **Note:** Professional C++ developers avoid `using namespace` in header files because it causes "namespace pollution"—essentially bringing back the very naming conflicts namespaces were meant to solve!

## Summary
Namespaces turn global name management from a **social problem** (hoping developers don't use the same name) into a **structural feature**. In C, you are forced to give functions long, ugly names. In C++, you can use short, intuitive names like `init()` and let the namespace provide the necessary context.
