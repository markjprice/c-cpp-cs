> **Prompt**: Please show me how C++ classes solve the struct plus functions problem in C.

In C, a `struct` and the functions that operate on it are "strangers" held together only by the programmer's discipline. C++ classes solve this by unifying **data** and **behavior** into a single entity, enforcing the "ownership" and "privacy" that C lacks.

Here is the breakdown of how classes transform the "Struct + Functions" mess into a cohesive object.

---

## 1. The C Way: "Disconnected Logic"
In C, the data and the logic are physically separated. The functions have to "reach into" the struct, and there is no way to stop a rogue function from corrupting the data.

```c
// The Data
struct BankAccount {
    double balance;
};

// The Behavior (somewhere else in the file)
void deposit(struct BankAccount *account, double amount) {
    if (amount > 0) {
        account->balance += amount;
    }
}

// The Problem: Nothing stops me from doing this:
// account.balance = -1000000.0; // Direct access bypasses logic!
```

---

## 2. The C++ Solution: "Encapsulation"
A C++ class wraps the data inside a "shell." It uses **access specifiers** (`public`, `private`) to dictate who can touch what.



```cpp
class BankAccount {
private:
    double balance = 0; // Private: No one can touch this directly!

public:
    void deposit(double amount) { // Public: This is the only way in
        if (amount > 0) {
            balance += amount;
        }
    }

    double getBalance() { return balance; }
};
```

### Why this is a massive upgrade:
* **The "Invariant" is Protected:** In C++, it is physically impossible for an outside function to set `balance` to a negative number. The class acts as a gatekeeper.
* **Reduced Argument Noise:** In C, every function needs a pointer to the struct (`struct BankAccount *acc`). In C++, the function is *part* of the object, so it implicitly knows which data it is working on via the `this` pointer.

---

## 3. Solving the "Initialization" Nightmare
In C, if you forget to call an `init()` function, your struct contains "garbage" memory, which leads to crashes. C++ solves this with **Constructors**.

| Feature | C Struct + Functions | C++ Classes |
| :--- | :--- | :--- |
| **Setup** | Manual (e.g., `init_account(&acc)`) | Automatic (Constructor) |
| **Cleanup** | Manual (e.g., `destroy_account(&acc)`) | Automatic (Destructor) |
| **Visibility** | Everything is public (Open) | Data is hidden (Private) |

### The Lifecycle Guarantee
When a C++ object is created, the constructor runs automatically. When it goes out of scope, the **destructor** runs automatically. This eliminates the "I forgot to free the memory" bug that plagues large C codebases.

## Summary
C++ classes don't just "organize" the code; they **protect** it. By moving functions inside the same scope as the data, the compiler can enforce rules about how that data is modified. This turns a "maintenance nightmare" of chasing global state into a "modular system" where each class is responsible for its own integrity.
