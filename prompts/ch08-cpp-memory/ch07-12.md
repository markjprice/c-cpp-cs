The "Rules" of C++ are architectural guidelines that ensure your objects manage memory safely and efficiently. They are directly tied to the concept of **Ownership**—the responsibility of a class to clean up the resources it allocates.

---

## 1. The Rule of Three (The C++98 Standard)
The Rule of Three states that if your class manages a resource (like heap memory) and you need to define one of these three, you likely need all three:

1.  **Destructor:** To free the memory when the object dies.
2.  **Copy Constructor:** To perform a **Deep Copy** of the resource.
3.  **Copy Assignment Operator:** To handle copying from an existing object (`a = b`).

### The "Why": Preventing Memory Chaos
If you have a class that uses `new` to allocate an array but you *don't* define a Copy Constructor, C++ will do a "Shallow Copy."



As shown above, both objects point to the same memory. When the first one is destroyed, it deletes the data. When the second one is destroyed, it tries to delete it again, causing a **Double Free Crash**.

---

## 2. The Rule of Five (The Modern C++11 Standard)
With the introduction of **Move Semantics**, we added two more rules to make code faster. Instead of duplicating data, we can "steal" it from temporary objects.

4.  **Move Constructor:** To transfer ownership of a resource.
5.  **Move Assignment Operator:** To transfer ownership via the `=` operator.

### Memory Ownership Transfer
Moving is an $O(1)$ operation. You simply copy the pointer address to the new object and set the old pointer to `nullptr`.



**Comparison of Ownership:**
* **Copying:** "I want my own identical version of your data." (Expensive)
* **Moving:** "I am taking your data because you are finished with it." (Instant)

---

## 3. Real-World Code Example
Here is a class that manages a heap-allocated buffer, implementing the Rule of Five.

```cpp
class DataBuffer {
    int* data;
    size_t size;

public:
    DataBuffer(size_t s) : size(s), data(new int[s]) {} // Constructor

    // 1. Destructor
    ~DataBuffer() { delete[] data; }

    // 2. Copy Constructor (Deep Copy)
    DataBuffer(const DataBuffer& other) : size(other.size), data(new int[other.size]) {
        std::copy(other.data, other.data + size, data);
    }

    // 3. Copy Assignment
    DataBuffer& operator=(const DataBuffer& other) {
        if (this != &other) {
            delete[] data;
            size = other.size;
            data = new int[size];
            std::copy(other.data, other.data + size, data);
        }
        return *this;
    }

    // 4. Move Constructor (Resource Stealing)
    DataBuffer(DataBuffer&& other) noexcept : data(other.data), size(other.size) {
        other.data = nullptr; // Object is now empty
        other.size = 0;
    }

    // 5. Move Assignment
    DataBuffer& operator=(DataBuffer&& other) noexcept {
        if (this != &other) {
            delete[] data;
            data = other.data;
            size = other.size;
            other.data = nullptr;
            other.size = 0;
        }
        return *this;
    }
};
```

---

## 4. The Modern Goal: The Rule of Zero
The Rule of Zero states that you should design your classes so that you **don't have to write any of these five functions.** By using standard containers like `std::vector` or `std::string` as members, the compiler-generated versions of the Rule of Five will work perfectly and automatically.



### Summary Table: Which Rule to Use?

| Scenario | Rule to Follow | Reason |
| :--- | :--- | :--- |
| **Simple Data** (int, float) | **Rule of Zero** | No manual memory to manage. |
| **STL Members** (vector, string) | **Rule of Zero** | The STL already handles its own Rule of Five. |
| **Raw Pointers** (new/delete) | **Rule of Five** | Necessary to prevent leaks and crashes. |
| **Unique Handles** (File, Socket) | **Rule of Five** | Ensures only one object owns the resource. |
