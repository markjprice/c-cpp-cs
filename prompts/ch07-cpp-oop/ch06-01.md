> **Prompt**: Please show me some common mistakes beginners make with constructors and destructors in C++.

Understanding the lifecycle of an object is where most C++ bugs are born. In C, you just had data; in C++, you have an object that has a "birth" (constructor) and a "death" (destructor).

Here are the most common mistakes beginners make during these two critical phases.

---

### 1. The "Most Vexing Parse"
This is a famous C++ trap where the compiler thinks you are declaring a function instead of creating an object.

**The Mistake:**
```cpp
class Timer {};

int main() {
    Timer t(); // You think you created a Timer called t.
               // The compiler thinks you declared a function named 't' 
               // that returns a Timer and takes no arguments.
}
```
* **The Fix:** Use curly braces (Uniform Initialization) or omit the parentheses.
    ```cpp
    Timer t{}; // Modern C++ (Safe)
    Timer t;   // Classic C++ (Safe)
    ```

---

### 2. Forgetting the Virtual Destructor
This is perhaps the most dangerous bug in C++ inheritance. If you delete a derived class object through a base class pointer, and the base class doesn't have a `virtual` destructor, the derived part of the object is never cleaned up.

**The Mistake:**
```cpp
class Base {
    ~Base() {} // NOT VIRTUAL
};

class Derived : public Base {
    int* data = new int[100];
    ~Derived() { delete[] data; }
};

Base* ptr = new Derived();
delete ptr; // ❌ BUG: Only ~Base() is called. 'data' is leaked!
```

* **The Fix:** If a class has even one virtual function, its destructor **must** be virtual.
    ```cpp
    virtual ~Base() = default;
    ```

---

### 3. Misunderstanding Initialization Order
In C++, members are initialized in the **order they are declared in the class**, not the order they appear in your constructor's initialization list.

**The Mistake:**
```cpp
class Buffer {
    int size;
    int* data;
public:
    // Beginner thinks 'size' is set first, then 'data' is allocated
    Buffer(int s) : data(new int[size]), size(s) {} 
    // ❌ BUG: 'data' is declared AFTER 'size' in the class? No.
    // If 'data' was declared first in the header, 'size' is still garbage 
    // when 'new int[size]' is called.
};
```
* **The Fix:** Always match the order in your initializer list to the order of declaration in the header.

---

### 4. Throwing Exceptions in a Destructor
Destructors are called during "stack unwinding" (when an exception is already looking for a handler). If a destructor throws *another* exception while one is already active, the program will immediately terminate (`std::terminate`).

**The Mistake:**
```cpp
~MyFile() {
    if (!closed) throw std::runtime_error("File not closed!"); // ❌ FATAL
}
```
* **The Fix:** Destructors should never throw. Catch all potential errors inside the destructor and handle them quietly (e.g., log them).

---

### 5. Manual Cleanup Instead of RAII
Beginners often write "cleanup" functions and call them manually, or use `delete` inside the constructor if something fails, leading to double-frees or leaks.

**The Mistake:**
```cpp
class Database {
    int* logs;
public:
    Database() {
        logs = new int[10];
        if (connection_fails()) {
            delete[] logs; // Trying to be helpful...
            throw "Error"; // ...but the destructor might run too!
        }
    }
};
```
* **The Fix:** Use **Smart Pointers** (`std::unique_ptr`). If the constructor throws, the smart pointer's destructor runs automatically, and you don't have to write a single `delete`.

---

### 6. The "Shallow Copy" Disaster
If your class manages a pointer and you don't write a **Copy Constructor**, C++ provides a default one that just copies the pointer address. Now two objects think they own the same memory.



**The Mistake:**
```cpp
{
    MyString s1("Hello");
    MyString s2 = s1; // Both s1 and s2 point to the same "Hello" on the heap.
} // ❌ CRASH: s2 is destroyed and frees memory. s1 is destroyed and tries to free the same memory!
```
* **The Fix:** Follow the **Rule of Three** (if you need a destructor, you need a copy constructor and copy assignment operator) or, better yet, the **Rule of Zero** (use `std::string` or `std::vector` so you don't have to manage memory at all).

---

### Summary Table: Constructor/Destructor Safety

| Feature | Beginner Error | Professional Practice |
| :--- | :--- | :--- |
| **Initialization** | Assignment in body `{ }` | Initializer list `: member(val)` |
| **Inheritance** | Normal destructor | `virtual` destructor in Base |
| **Cleanup** | `delete` in destructor | Use Smart Pointers (RAII) |
| **Copying** | Relying on default copy | Disable copy or write Deep Copy |
| **Failure** | Return error codes | Throw an exception from Constructor |
