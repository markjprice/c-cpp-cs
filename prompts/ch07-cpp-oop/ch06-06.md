> **Prompt**: Please show me a step-by-step example of how virtual functions enable runtime polymorphism in C++.

To understand runtime polymorphism, think of it as a **"Late Binding"** contract. The compiler doesn't decide which code to run when it sees a function call; instead, it leaves a note that says, "Wait until the program is running, look at the actual object, and then decide."

Here is a step-by-step breakdown of how this works in practice.

---

### Step 1: Define the Base Class (The Interface)
The `virtual` keyword is the signal to the compiler that this function should be entered into the **vtable** (Virtual Method Table).

```cpp
class Shape {
public:
    // Virtual function: can be overridden by children
    virtual void draw() {
        std::cout << "Drawing a generic shape." << std::endl;
    }

    // Always provide a virtual destructor in base classes!
    virtual ~Shape() = default;
};
```

### Step 2: Create Derived Classes (The Implementations)
The children classes "override" the base function. Even though they have the same name, they reside at different memory addresses.

```cpp
class Circle : public Shape {
public:
    void draw() override { // 'override' is optional but recommended
        std::cout << "Drawing a Circle ◯" << std::endl;
    }
};

class Square : public Shape {
public:
    void draw() override {
        std::cout << "Drawing a Square □" << std::endl;
    }
};
```

### Step 3: Use Base Class Pointers (The Mechanism)
For polymorphism to work, you must access the objects through a **pointer** or **reference** to the base class.

```cpp
void render(Shape* s) {
    // At compile time, the compiler only knows 's' is a Shape.
    // It doesn't know if it's a Circle or a Square.
    s->draw(); 
}
```

### Step 4: Execution at Runtime (The Polymorphism)
When the code runs, the program follows the object's internal pointer to find its specific implementation.

```cpp
int main() {
    Circle c;
    Square s;

    // We pass different types to the same function
    render(&c); // Output: Drawing a Circle ◯
    render(&s); // Output: Drawing a Square □

    return 0;
}
```

---

## How It Works Behind the Scenes

When you call `s->draw()`, the CPU performs a hidden "three-step jump" known as the **vtable lookup**:

1.  **Follow the Object Pointer:** The CPU looks at the `Shape* s`.
2.  **Look up the vtable:** Every object of a class with virtual functions has a hidden pointer (the **vptr**) that points to a table of function addresses (the **vtable**).
3.  **Execute the Address:** It finds the address for `draw()` in that table and jumps to it.



| Component | Responsibility |
| :--- | :--- |
| **`virtual` keyword** | Tells the compiler to create a vtable for this class. |
| **vptr (Pointer)** | A hidden member in every object pointing to the vtable. |
| **vtable (Table)** | A static array of function pointers for that specific class. |
| **Dynamic Dispatch** | The act of looking up the function at runtime. |



### Why this is powerful
You can add a `Triangle` class later without ever touching or recompiling the `render()` function. As long as `Triangle` inherits from `Shape`, the `render()` function will "just work." This is the core of the **Open-Closed Principle** (software should be open for extension but closed for modification).
