> **Prompt**: Please explain the difference between composition and inheritance with real-world C++ examples and when each should be used.

The choice between **Inheritance** and **Composition** is a fundamental architectural decision in C++. While Inheritance was the "golden child" of early Object-Oriented Programming, modern C++ philosophy strongly leans toward **Composition** (often summarized as "Favor composition over inheritance").

---

## 1. Inheritance: The "Is-A" Relationship
Inheritance means one class is a specialized version of another. It allows the derived class to inherit the data and behavior of the base class.

* **Relationship:** "A Dog **is an** Animal."
* **Mechanism:** Derived classes extend the base class.
* **Key Feature:** Supports **Polymorphism** (using a `Base*` to point to a `Derived` object).

### C++ Example: Game Entities
```cpp
class Entity {
public:
    virtual void update() = 0; // Pure virtual
    virtual ~Entity() = default;
};

class Player : public Entity {
public:
    void update() override { /* Move player based on input */ }
};

class NPC : public Entity {
public:
    void update() override { /* Move based on AI */ }
};
```
**When to use:** Use inheritance only when you need a true hierarchical relationship and want to treat different objects as the same type (Polymorphism).



---

## 2. Composition: The "Has-A" Relationship
Composition means a class is built using other classes as building blocks. Instead of *being* something else, it *holds* something else.

* **Relationship:** "A Car **has an** Engine."
* **Mechanism:** Objects are members (fields) of another class.
* **Key Feature:** High **Flexibility** and low "coupling" (classes don't depend as heavily on each other's internal logic).

### C++ Example: Building a Character
```cpp
class Health {
public:
    void takeDamage(int amount) { hp -= amount; }
private:
    int hp = 100;
};

class Inventory {
public:
    void addItem(std::string item) { items.push_back(item); }
private:
    std::vector<std::string> items;
};

class Character {
private:
    // Composition: Character HAS health and an inventory
    Health health;
    Inventory inventory;
public:
    void hit(int damage) { health.takeDamage(damage); }
};
```
**When to use:** Use composition for almost everything else. It allows you to "mix and match" behaviors without creating a messy, deep inheritance tree.



---

## 3. Comparison at a Glance

| Feature | Inheritance (Is-A) | Composition (Has-A) |
| :--- | :--- | :--- |
| **Coupling** | **Tight:** Changes in Base affect all Derived. | **Loose:** Changes in components are isolated. |
| **Flexibility** | **Static:** Defined at compile-time. | **Dynamic:** Can be changed or swapped easily. |
| **Encapsulation** | **Weak:** Derived class often sees Base internals. | **Strong:** Only interacts with public interfaces. |
| **Testing** | Harder (requires testing the whole hierarchy). | Easier (components can be tested individually). |

---

## 4. The "Liskov Substitution" Test
If you are unsure which to use, ask yourself: **"Can I use the child object anywhere the parent object is expected without the program breaking?"**

* **Example:** A `Square` inherits from `Rectangle`. 
    * A `Rectangle` has `set_width` and `set_height`. 
    * If you call `set_width(10)` and `set_height(20)` on a `Square`, the rules of a square are broken. 
    * **Conclusion:** A Square is **not** a Rectangle in programming terms. You should use composition or a different interface.

---

## 5. Why Modern C++ Favors Composition
Deep inheritance trees lead to the **"Fragile Base Class"** problem. If you change a single variable in a Base class at the top of a 10-level hierarchy, you might accidentally break 100 unrelated Derived classes.

Composition prevents this by keeping classes small, focused, and independent. In C++, this also plays nicely with **RAII**; when the "Owner" class is destroyed, all its composed "Member" objects are automatically cleaned up as well.
