> **Prompt**: What problems does RAII solve in C++ and how is it used in modern code?

At its core, **RAII (Resource Acquisition Is Initialization)** is a programming idiom that links the "lifespan" of a resource (like memory, file handles, or mutexes) to the "lifespan" of a local object.

In C, you manage resources **imperatively** (you tell the computer when to start and when to stop). In C++, you manage them **declaratively** (you define the scope, and the language handles the cleanup).

---

## 1. Problems Solved by RAII

### A. The "Leaky Return" Problem
In languages without RAII, you must remember to free every resource before every possible `return` statement. If a function has five different exit points, you need five cleanup calls.

**The Bug (C style):**
```c
void process() {
    FILE* f = fopen("data.txt", "r");
    if (!f) return;

    if (error_condition) {
        // Oops! Forgot to call fclose(f) here.
        return; 
    }

    fclose(f);
}
```

### B. The Exception Safety Nightmare
When an exception is thrown in C++, the execution jumps immediately to the nearest `catch` block. Any manual cleanup code located after the line that threw the exception is skipped, leading to a "zombie" resource that is never released.



### C. Double-Frees and Dangling Pointers
By tying the resource to a single object's destructor, you ensure that the resource is freed exactly once: when the object goes out of scope. This prevents the common mistake of accidentally freeing the same memory twice or using memory after it has been deleted.

---

## 2. How RAII Works Under the Hood
RAII relies on two guaranteed features of C++:
1.  **Constructors:** Run exactly when the object is created.
2.  **Destructors:** Run exactly when the object goes out of scope (even if an exception occurs).



---

## 3. RAII in Modern C++ Code
Modern C++ (C++11 and later) has largely replaced raw pointers and manual cleanup with specialized RAII wrappers.

### Smart Pointers (`std::unique_ptr` and `std::shared_ptr`)
Instead of `new` and `delete`, we use smart pointers. When the pointer object is destroyed, it automatically deletes the memory it points to.
```cpp
{
    auto ptr = std::make_unique<MyClass>(); 
    // Do work...
} // Memory is freed AUTOMATICALLY here.
```

### Containers (`std::vector`, `std::string`)
You don't need to manage arrays or null-terminated strings. The container manages a heap-allocated buffer. When the vector goes out of scope, it destroys its elements and frees the buffer.

### Thread Synchronization (`std::lock_guard`)
One of the most common RAII uses is managing mutexes. If you forget to unlock a mutex, your program deadlocks.
```cpp
std::mutex mtx;

void safe_function() {
    std::lock_guard<std::mutex> lock(mtx); // Mutex is locked here
    // If this code throws an exception, the mutex is STILL unlocked!
} // Mutex is unlocked AUTOMATICALLY here.
```

### File I/O (`std::ifstream`)
The `std::ifstream` constructor opens the file, and the destructor closes it. 
```cpp
{
    std::ifstream file("config.json");
    // read file...
} // File handle is closed AUTOMATICALLY here.
```

---

## 4. Summary: The "Rule of Zero"
The goal of RAII in modern code is to follow the **Rule of Zero**: your business logic classes should not have to define destructors, copy constructors, or move constructors at all. Instead, they should use RAII objects (like `std::string` or `std::unique_ptr`) as member variables, which handle their own cleanup automatically.

| Resource | Old Way (Manual) | Modern RAII Way |
| :--- | :--- | :--- |
| **Heap Memory** | `malloc` / `free` | `std::unique_ptr` / `std::vector` |
| **Files** | `fopen` / `fclose` | `std::fstream` |
| **Mutexes** | `pthread_mutex_lock` | `std::lock_guard` |
| **Strings** | `char*` | `std::string` |
