> **Prompt**: Compare abstract classes in C++ with interfaces in C#.

While C++ and C# both use these structures to define "contracts" for classes, their implementation reflects the different philosophies of the two languages. C# has a dedicated `interface` keyword and a separate `abstract` class concept, while C++ uses **Abstract Classes** to serve both purposes.

---

## 1. The Core Definitions

### C++: Abstract Classes
In C++, a class becomes abstract if it contains at least one **pure virtual function**. 
* **Syntax:** `virtual void func() = 0;`
* **Purpose:** It acts as a base class that cannot be instantiated. It can contain both implementation (member variables and defined functions) and pure signatures.

### C#: Interfaces vs. Abstract Classes
C# makes a hard distinction:
* **Abstract Class:** A base class that can have state (fields) and partial implementation. A class can only inherit from **one** abstract class.
* **Interface:** A strict contract. Historically, interfaces could have *no* implementation (though modern C# allows default implementations). A class can implement **multiple** interfaces.

---

## 2. Key Technical Differences

| Feature | C++ Abstract Class | C# Interface | C# Abstract Class |
| :--- | :--- | :--- | :--- |
| **Keyword** | None (uses `= 0`) | `interface` | `abstract` |
| **Multiple Inheritance**| **Supported.** You can inherit from many. | **Supported.** | **Not Supported.** (Single inheritance only). |
| **Data Members** | Can have variables/fields. | No fields allowed. | Can have variables/fields. |
| **Access Modifiers** | Supports `public`, `private`, `protected`. | Default is `public`. | Supports all modifiers. |
| **Constructor** | Can have a constructor. | Cannot have a constructor. | Can have a constructor. |

---

## 3. Real-World Syntax Comparison

### C++ (The "Hybrid" Approach)
In C++, if you want something to behave like a pure interface, you simply create an abstract class with only pure virtual functions and no data members.

```cpp
class IShape {
public:
    virtual void draw() = 0; // Pure virtual
    virtual ~IShape() = default; 
};

class Circle : public IShape {
public:
    void draw() override { /* ... */ }
};
```

### C# (The "Explicit" Approach)
C# forces you to choose between a "contract" and a "partial implementation."

```csharp
// The Contract
public interface IShape {
    void Draw(); 
}

// The Partial Implementation
public abstract class BaseShape {
    public string Color { get; set; } // Can have state
    public abstract void Draw();
}
```



---

## 4. Philosophical Differences

### Multiple Inheritance vs. Interfaces
The biggest difference is how they handle "combining" behaviors. 
* **C++** allows **Multiple Inheritance**. You can inherit from three different abstract classes. This is powerful but can lead to the "Diamond Problem," where the compiler doesn't know which parent's version of a function to use.
* **C#** avoids the Diamond Problem by prohibiting multiple class inheritance. It uses **Interfaces** to allow a class to "wear many hats" (e.g., a `User` class can be `IDeletable`, `IAuthenticatable`, and `ISerializable`) without the risks of complex inheritance trees.



### Memory and Vtables
* In **C++**, an abstract class works through a standard **vtable**. The pointer to the vtable is stored directly in the object.
* In **C#**, calling a method through an interface often involves an **Interface Method Table (IMT)** lookup. This is slightly more complex than a standard virtual call because a class might implement multiple interfaces, each requiring its own mapping.

---

## 5. When to Use Which?

* **Use a C# Interface** when you want to define a behavior that can be shared by totally unrelated classes (e.g., `ICloneable`).
* **Use a C# Abstract Class** when you want to provide a common base with shared code for a group of closely related classes.
* **In C++**, use an abstract class with no data (an "Interface") for high-level architectural boundaries, and an abstract class with data for sharing implementation details among derived types.
